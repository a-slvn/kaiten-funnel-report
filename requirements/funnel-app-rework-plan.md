# План доработки приложения Funnel Report

## Назначение

Этот документ фиксирует согласованный план доработки текущего приложения Funnel Report на основе обсуждения и фидбека CPO.

---

## Финальные бизнес-правила

1. Обычные колонки доски считаются этапами воронки.
2. Колонка типа `done` считается колонкой `Оплачено`.
3. Колонка сразу после `done` считается колонкой `Проиграно`.
4. Если поле суммы в настройках доски не задано, отчёт строится только по количеству сделок.
5. В режиме без суммы денежные данные в интерфейсе не отображаются.
6. Если поле суммы не задано, пользователю показывается подсказка с предложением выбрать поле суммы для более полного отчёта.

---

## Целевая модель отчёта

- Отчёт открывается сразу, без обязательной предварительной настройки.
- Воронка строится автоматически по структуре доски.
- Базовый режим работы отчёта: `count`.
- Расширенный режим: `amount`, если в доске указано поле суммы.
- Пользовательская настройка в первой версии должна сводиться к выбору поля суммы, а не к ручной разметке колонок.

---

## План работ

### Must

#### 1. Убрать стоп-экран первого открытия

- Удалить сценарий, в котором отчёт не отображается до подтверждения настройки.
- Сделать автоматическое открытие графика при первом заходе.

#### 2. Реализовать автоконфигурацию воронки

- Построить последовательность этапов автоматически:
  - все обычные колонки;
  - колонка `done` как `Оплачено`;
  - колонка после `done` как `Проиграно`.
- Исключить обязательную ручную настройку ролей колонок из основного сценария.

#### 3. Переписать модель конфигурации отчёта

- Убрать текущую ручную setup-модель как обязательный источник истины.
- Хранить только минимально необходимую конфигурацию:
  - выбранное поле суммы;
  - служебные параметры автоконфигурации;
  - возможные override-настройки для будущих итераций.

#### 4. Ввести два режима данных

- `count` по умолчанию.
- `amount`, если поле суммы указано.
- Зафиксировать `metric_mode` как часть data contract.

#### 5. Переписать расчёты отчёта

- Считать воронку по новой последовательности этапов.
- Явно учитывать `Оплачено` и `Проиграно`.
- Если поле суммы отсутствует:
  - не считать денежные показатели;
  - не возвращать денежные агрегаты в UI-модель.

#### 6. Переписать визуализацию под условные денежные данные

- В графике скрывать денежные подписи в режиме `count`.
- В таблице скрывать денежные колонки, если суммы нет.
- KPI-карточки делать адаптивными:
  - в `count` показывать только количественные метрики;
  - в `amount` добавлять денежные показатели.

#### 7. Добавить alert про отсутствие поля суммы

- Показать баннер над графиком, если поле суммы не задано.
- Текст баннера:
  - отчёт построен по количеству сделок;
  - выберите поле суммы, чтобы увидеть полный отчёт.
- CTA из баннера должен вести в настройку выбора поля суммы.

#### 8. Сделать настройку поля суммы

- Добавить простой сценарий выбора numeric field для суммы.
- Не возвращать сложную ручную настройку ролей колонок как основной UX.

---

### Should

#### 9. Починить фильтры

- Связать фильтры боковой панели с реальным state данных.
- Сделать кнопку `Показать` рабочей.
- Убедиться, что даты и фильтры реально влияют на построение отчёта.

#### 10. Удалить дублирующие конфиг-компоненты

- Упростить конфигурационный слой.
- Удалить или объединить старые компоненты настройки, оставив один понятный сценарий.

#### 11. Подготовить backend/API contract

- Backend должен отдавать:
  - stages;
  - `metric_mode`;
  - `amount_field_id`;
  - причину отсутствия денежного режима;
  - предупреждения для интерфейса.

---

### Later

#### 12. Добавить пользовательские override-настройки

- Дать возможность точечно переопределять автоконфигурацию для нестандартных кейсов.
- Не делать этот сценарий обязательным.

#### 13. Добавить продуктовую аналитику

- Зафиксировать события:
  - автоконфигурация применена;
  - показан alert об отсутствии суммы;
  - пользователь выбрал поле суммы.

---

## Рекомендуемый порядок реализации

1. Убрать stop-screen.
2. Реализовать auto-build воронки по колонкам доски.
3. Ввести `count` как режим по умолчанию.
4. Переписать chart, table и KPI под условное отображение денег.
5. Добавить alert и выбор поля суммы.
6. Починить фильтры.
7. Почистить старый setup-код.
8. Подключить backend contract.

---

## Главный принцип реализации

Основная доработка приложения заключается не в расширении ручной настройки, а в переходе на:

- автоматическое построение по структуре доски;
- `count-first` UX;
- опциональное обогащение отчёта через выбор поля суммы.
