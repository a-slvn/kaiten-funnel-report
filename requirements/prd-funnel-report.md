## Общее описание

### Проблема

Руководители отделов продаж, использующие Kaiten как CRM, не имеют инструмента для визуального анализа прохождения сделок через этапы воронки. Сейчас для понимания картины нужно вручную считать карточки в колонках доски, что:
- Занимает 15-30 минут вместо 10 секунд
- Не даёт исторической картины (видно только текущее состояние доски)
- Не показывает конверсию между этапами
- Не позволяет сравнить периоды
- Не масштабируется при 100+ сделках

### Решение

Отчёт «Воронка продаж» -- интерактивная визуализация прохождения сделок через этапы воронки. Воронка строится по всем доскам пространства как единый поток, объединяя колонки всех досок в одну последовательность этапов. Отчёт показывает количество сделок, сумму, конверсию и среднее время на каждом этапе, поддерживает фильтрацию, drill-down и сравнение периодов.

Отчёт следует паттерну **Best Guess**, принятому в графиках Kaiten (референс -- кумулятивная диаграмма): график строится автоматически при открытии, без обязательной предварительной настройки. Пользователь видит результат сразу и при необходимости корректирует конфигурацию.

### Целевая аудитория

| Роль | Размер сегмента | Частота использования | Ключевая задача |
|---|---|---|---|
| Руководитель отдела продаж | Основная | 3-5 раз в неделю | Контроль воронки, поиск узких мест |
| Собственник / CEO | Вторичная | 1-2 раза в неделю | Оценка здоровья бизнеса, динамика |
| Менеджер по продажам | Вторичная | 1-2 раза в неделю | Анализ своих показателей |
| Аналитик | Вторичная | 2-3 раза в неделю | Детальный анализ, подготовка отчётов |

### Ключевые сценарии использования

1. **Ежедневный мониторинг** — руководитель открывает воронку, видит текущее состояние, замечает просадку конверсии на этапе «Коммерческое предложение».
2. **Еженедельный отчёт** — руководитель сравнивает текущую неделю с предыдущей, видит delta по конверсии и сумме на каждом этапе.
3. **Разбор проблемного этапа** — руководитель делает drill-down в этап с низкой конверсией, видит список зависших сделок, переходит в карточки для действий.
4. **Анализ по менеджерам** — руководитель фильтрует воронку по конкретному менеджеру, чтобы оценить его эффективность.
5. **Подготовка отчёта для CEO** — аналитик настраивает фильтры, экспортирует визуализацию и данные для включения в презентацию.

## Точка входа

### Расположение в интерфейсе

Отчёт располагается в разделе **«Отчёты»** главного меню Kaiten.

| Элемент | Описание |
|---|---|
| Меню | Главное меню -> «Отчёты» |
| Пункт | «Воронка продаж» с иконкой FunnelOutlined |
| URL | `/reports/sales-funnel?space_id={id}` |
| Доступ | Пользователи с правами на просмотр пространства |

### Навигация

1. Пользователь кликает «Отчёты» в главном меню Kaiten
2. В списке отчётов видит пункт «Воронка продаж»
3. При клике открывается страница отчёта в контексте текущего пространства
4. Если для пространства есть сохранённая конфигурация -- загружается отчёт с этой конфигурацией
5. Если конфигурация не сохранена -- запускается алгоритм Best Guess: система автоматически определяет конфигурацию воронки по всем доскам пространства и строит график сразу
6. Если Best Guess обнаруживает неоднозначность -- над графиком показывается алерт с предложением скорректировать настройки (график при этом уже отображается)

## Функциональные требования

### 1. Конфигурация воронки

Конфигурация определяет, откуда берутся данные и как строится воронка. Конфигурация бывает двух типов: **автоматическая** (Best Guess) и **пользовательская** (после ручной корректировки). Пользовательская конфигурация сохраняется между сессиями.

#### 1.1. Автоконфигурация по пространству (Best Guess)

При первом открытии отчёта (или при отсутствии сохранённой конфигурации) система автоматически строит конфигурацию воронки по алгоритму Best Guess. Подробное описание алгоритма: `research/best-guess-algorithm.md`.

**Входные данные:** `space_id` -- идентификатор текущего пространства.

**Что определяется автоматически:**

| Параметр | Как определяется | Описание |
|---|---|---|
| Доски | Все неархивные доски пространства | Порядок: сверху вниз, слева направо (как в кумулятивной диаграмме) |
| Этапы (колонки) | Колонки всех досок, кроме `column_type = 'done'` | Порядок: доска 1 (все колонки по `sort_order`) -> доска 2 -> ... -> доска N |
| Won-колонки | Последняя `done`-колонка по глобальному порядку | Сделки, дошедшие до этой колонки, считаются выигранными |
| Lost-колонки | Остальные `done`-колонки | Сделки в этих колонках считаются проигранными |
| Поле суммы | Общее числовое поле с одинаковым именем на всех досках | Если нет общего поля -- fallback на количество карточек |
| Режим метрики | `amount` (если поле суммы определено) или `count` (fallback) | По умолчанию `count` |

**Жизненный цикл конфигурации:**

1. **Первое открытие** -- Best Guess вычисляет конфигурацию на лету (не сохраняется в БД)
2. **Пользователь доволен** -- работает на auto-config при каждом открытии
3. **Пользователь корректирует** -- открывает настройки, изменяет, нажимает «Сохранить» -- конфигурация сохраняется в БД, Best Guess больше не запускается
4. **Сброс к Best Guess** -- пользователь может удалить сохранённую конфигурацию и вернуться к автоопределению

**Уровень уверенности (`confidence`):**

| Уровень | Когда | Алерт |
|---|---|---|
| `high` | 2 done-колонки, общее поле суммы | Нет |
| `medium` | 1 или 3+ done-колонок, или несколько общих полей суммы | Info/Warning |
| `low` | 0 done-колонок (последняя колонка назначена как won) | Warning |

#### 1.2. Определение этапов воронки

Этапы воронки определяются автоматически на основе колонок всех досок пространства.

| Параметр | Описание |
|---|---|
| Источник | Колонки всех неархивных досок пространства |
| Исключения | Колонки с `column_type = 'done'` (обрабатываются как won/lost) |
| Порядок по умолчанию | Доски: `row_sort_order` ASC, `sort_order` ASC; колонки внутри доски: `sort_order` ASC |
| Одноимённые колонки | Если на разных досках есть колонки с одинаковым названием -- к ним добавляется суффикс с названием доски: «В работе (Лиды)», «В работе (Сделки)» |

**Ручная корректировка (через панель настроек):**

| Поле | Тип | Ограничения | Назначение |
|---|---|---|---|
| Этапы (колонки) | Sortable checkbox list | Минимум 2 включённых; drag-and-drop для изменения порядка | Позволяет исключить нерелевантные колонки и изменить порядок |
| Доски | Checkbox list | Минимум 1 включённая доска | Позволяет исключить нерелевантные доски (все их колонки исключаются) |
| Won-колонки | Select | Минимум 1 | Позволяет переопределить автоопределённые won-колонки |
| Lost-колонки | MultiSelect | Опционально | Позволяет переопределить lost-колонки |

**Примечание (v2):** В будущей итерации при исключении промежуточной колонки её данные суммируются в предыдущий видимый этап (F-006). В первой версии исключение колонки полностью убирает её данные из расчётов.

#### 1.3. Поле суммы сделки и режим метрики

| Поле | Тип | Обязательное | Ограничения | Назначение |
|---|---|---|---|---|
| Режим метрики | ToggleButton (`count` / `amount`) | Да | По умолчанию `count` | Определяет основную метрику воронки |
| Поле суммы | Select (выпадающий список) | Нет | Числовые кастомные поля всех досок пространства; пункт «Не выбрано» | Определяет, какое поле использовать для расчёта сумм |

**Логика fallback:**

| Ситуация | Режим | Поведение |
|---|---|---|
| Поле суммы определено (автоматически или вручную) | Доступны `count` и `amount`; по умолчанию `count` | Переключатель «Количество / Сумма» активен |
| Поле суммы не определено (нет числовых полей или поля несовместимы) | Только `count` | Переключатель «Сумма» задизейблен с тултипом: «Настройте числовое поле для суммы сделки» |

Если поле суммы не выбрано, воронка строится по количеству карточек. Метрики «Сумма сделок» и «Средний чек» не отображаются.

### 2. Фильтры

Фильтры расположены в горизонтальной панели над диаграммой. При применении фильтра воронка перестраивается.

#### 2.1. Период

| Поле | Тип | Обязательное | Значение по умолчанию | Ограничения |
|---|---|---|---|---|
| Период | Select + DateRangePicker | Да | Текущий месяц | Начальная дата <= конечной даты |

**Предустановленные значения:**

| Значение | Код | Описание |
|---|---|---|
| Сегодня | `today` | С начала текущего дня |
| Последние 7 дней | `last_7_days` | 7 календарных дней до сегодня |
| Текущий месяц | `current_month` | С 1-го числа текущего месяца |
| Предыдущий месяц | `previous_month` | Полный предыдущий месяц |
| Текущий квартал | `current_quarter` | С начала текущего квартала |
| Текущий год | `current_year` | С 1 января текущего года |
| Кастомный | `custom` | Произвольный диапазон дат |

**Логика применения периода:** В воронку попадают карточки, которые находились на включённых этапах (колонках) в рамках выбранного периода. Основание — история перемещений карточек. Карточка учитывается на этапе, если она переместилась на этот этап в пределах периода ИЛИ находилась на нём на начало периода.

#### 2.2. Ответственный

| Поле | Тип | Обязательное | Значение по умолчанию | Ограничения |
|---|---|---|---|---|
| Ответственный | MultiSelect с поиском | Нет | Все (не выбраны) | Список участников всех досок пространства; поиск по имени; «Выбрать всех» / «Сбросить» |

Ответственный определяется по полю «Ответственный» (owner) карточки.

#### 2.3. Теги

| Поле | Тип | Обязательное | Значение по умолчанию | Ограничения |
|---|---|---|---|---|
| Теги | MultiSelect с поиском | Нет | Все (не выбраны) | Список тегов карточек на досках пространства; логика «ИЛИ» -- карточка проходит, если содержит хотя бы один выбранный тег |

#### 2.4. Сумма сделки

| Поле | Тип | Обязательное | Значение по умолчанию | Ограничения |
|---|---|---|---|---|
| Сумма от | NumberInput | Нет | Пусто | >= 0; целое или дробное |
| Сумма до | NumberInput | Нет | Пусто | >= значения «от» (если заполнено) |

Фильтр доступен только если поле суммы определено (автоматически через Best Guess или вручную в настройках).

#### 2.5. Панель фильтров — общее поведение

| Элемент | Описание |
|---|---|
| Индикатор фильтров | Бейдж с количеством активных фильтров (кроме периода) |
| Кнопка «Сбросить» | Появляется при наличии хотя бы одного активного фильтра (кроме периода); сбрасывает все фильтры, период остаётся |
| Применение | Фильтры применяются мгновенно при изменении (без кнопки «Применить») |
| Сохранение | Выбранные фильтры не сохраняются между сессиями (только настройки воронки сохраняются) |

### 3. Визуализация воронки

#### 3.1. Основной вид — Funnel Chart (трапецеидальная диаграмма)

Классическая воронка, где ширина каждого этапа пропорциональна количеству сделок (или сумме — переключается).

| Параметр | Описание |
|---|---|
| Ориентация | Вертикальная (сверху вниз) |
| Форма | Трапецеидальные секции, ширина пропорциональна значению |
| Пропорциональность | По количеству сделок (`count`, по умолчанию) или по сумме (`amount`) -- определяется переключателем `metricMode` |
| Интерактивность | Hover — подсветка секции + tooltip; Click — drill-down |

**Данные на каждом этапе (секции):**

| Метрика | Расположение | Формат |
|---|---|---|
| Название этапа | Слева от секции | Текст |
| Количество сделок | Внутри секции | Число (например, «147 сделок») |
| Сумма сделок | Внутри секции | Число с разделителями (например, «12 450 000 руб.»); только в режиме `amount` |
| Конверсия от предыдущего | Между секциями | Процент со стрелкой (например, «72% -->») |
| Среднее время | Tooltip при hover | «Среднее время: 4.2 дня» |
| Средний чек | Tooltip при hover | «Средний чек: 85 000 руб.»; только в режиме `amount` |

**Цветовая схема:**

| Элемент | Цвет |
|---|---|
| Секции воронки | Градиент от основного цвета (100% opacity наверху) к светлому (40% opacity внизу) |
| Конверсия >= 70% | Зелёный текст |
| Конверсия 40-69% | Оранжевый текст |
| Конверсия < 40% | Красный текст |

#### 3.2. Альтернативный вид — Bar Chart (столбчатая диаграмма)

| Параметр | Описание |
|---|---|
| Ориентация | Горизонтальная (этапы по оси Y, значения по оси X) |
| Данные | Количество сделок на каждом этапе |
| Подписи | Название этапа, количество, конверсия |
| Интерактивность | Hover — tooltip с полными метриками; Click — drill-down |

#### 3.3. Альтернативный вид — Таблица

| Столбец | Тип | Описание |
|---|---|---|
| Этап | Текст | Название колонки (этапа) |
| Кол-во сделок | Число | Количество карточек на этапе |
| Сумма | Число (форматированное) | Сумма по полю суммы; прочерк если `metricMode = 'count'` или поле суммы не определено |
| Средний чек | Число (форматированное) | Сумма / Количество; прочерк если `metricMode = 'count'` или поле суммы не определено |
| Конверсия (от пред.) | Процент | N этапа / N предыдущего * 100%; для первого этапа — прочерк |
| Конверсия (общая) | Процент | N этапа / N первого этапа * 100% |
| Среднее время | Число (дни) | Среднее время нахождения на этапе в днях |

Таблица поддерживает сортировку по любому столбцу.

#### 3.4. Переключатель видов

| Элемент | Описание |
|---|---|
| Расположение | Правый верхний угол области визуализации |
| Тип | ToggleButtonGroup (3 кнопки-иконки) |
| Иконки | Funnel / BarChart / Table |
| Поведение | Переключение без перезагрузки данных; выбранный вид сохраняется в localStorage |

#### 3.5. Переключатель «Количество / Сумма» (metric_mode)

Переключатель определяет основную метрику, по которой строится воронка. Это ключевой элемент управления, тесно связанный с алгоритмом Best Guess и fallback-логикой.

| Элемент | Описание |
|---|---|
| Расположение | Рядом с переключателем видов |
| Тип | ToggleButton (2 варианта) |
| Значения | «Количество» (`count`) / «Сумма» (`amount`) |
| По умолчанию | «Количество» (`count`) |
| Влияние | Определяет, по какой метрике строится пропорциональность воронки и основные числа |
| Доступность «Суммы» | Активна только если поле суммы определено (автоматически через Best Guess или вручную в настройках) |

**Поведение при недоступности «Суммы»:**

| Элемент | Описание |
|---|---|
| Состояние кнопки | Визуально задизейблена (серая, некликабельная) |
| Тултип при hover | «Настройте числовое поле для суммы сделки» (если нет полей) или «На досках пространства настроены разные числовые поля» (если конфликт) |
| Текущий режим | Принудительно `count` |

**Различия в отображении по режимам:**

| Элемент | `count` | `amount` |
|---|---|---|
| Основная цифра на этапе | Количество сделок (например, «147 сделок») | Сумма сделок (например, «12 450 000 руб.») |
| Пропорциональность секций | По количеству | По сумме |
| Средний чек | Не отображается | В tooltip при hover |
| Конверсия | Всегда по количеству сделок (не меняется) | Всегда по количеству сделок (не меняется) |

#### 3.6. Пустое состояние

Если по текущим фильтрам нет данных:

| Элемент | Описание |
|---|---|
| Иконка | EmptyFunnelOutlined (пустая воронка, 64px, серый) |
| Заголовок | «Нет данных для отображения» |
| Подпись | «Попробуйте изменить фильтры или выбрать другой период» |
| Кнопка | «Сбросить фильтры» (если фильтры активны) |

#### 3.7. Алерты автоконфигурации (Best Guess)

Алерты информируют пользователя о неоднозначностях в автоматически определённой конфигурации. Алерт **не блокирует** отображение графика (кроме ошибок) -- воронка всегда строится и показывается.

**Каталог алертов:**

| Код | Тип | Сообщение | Действие | Условие показа |
|-----|-----|-----------|----------|----------------|
| `NO_BOARDS` | error | На этом пространстве нет досок. Создайте доску, чтобы построить воронку. | -- | 0 досок в пространстве |

| `SINGLE_DONE_COLUMN` | info | Найдена одна завершающая колонка. Она используется как "Выигран". Если у вас есть отдельная колонка для проигранных сделок, укажите её в настройках. | Настроить | 1 done-колонка |

**Правила показа алертов:**

| Правило | Описание |
|---|---|
| Алерты `error` | Блокируют отображение графика. Вместо графика -- сообщение об ошибке |
| Алерты `warning` | Отображаются над графиком в виде баннера. График строится и отображается |
| Алерты `info` | Отображаются в сворачиваемом блоке над графиком. График строится |
| Максимум одновременно | Не более 2 алертов. Если больше -- показываются 2 самых приоритетных (error > warning > info). Остальные доступны по ссылке «ещё N» |
| Скрытие | Алерт скрывается, когда пользователь нажал «Настроить» и сохранил конфигурацию |
| Dismiss | Пользователь может закрыть алерт (dismiss). В рамках сессии он не показывается повторно. При следующем открытии отчёта -- показывается снова |
| Пользовательский конфиг | Если конфигурация сохранена пользователем (`auto_generated = false`) -- алерты Best Guess не показываются |

### 5. Сравнение периодов

Режим сравнения позволяет видеть разницу между двумя временными периодами.

#### 5.1. Включение режима

| Элемент | Описание |
|---|---|
| Кнопка | «Сравнить периоды» рядом с фильтром периода |
| Поведение | При клике появляется второй date picker для выбора периода сравнения |
| Выключение | Кнопка «X» рядом со вторым периодом; возврат к одиночному виду |

#### 5.2. Выбор периода сравнения

| Поле | Тип | Значение по умолчанию | Ограничения |
|---|---|---|---|
| Период сравнения | Select + DateRangePicker | Предыдущий период той же длительности | Те же предустановленные значения, что и для основного периода + «Предыдущий период» (авто) |

**«Предыдущий период»** — автоматически подбирается период той же длительности, непосредственно предшествующий основному. Например: если основной период — 1-31 января, то период сравнения — 1-31 декабря.

#### 5.3. Визуализация сравнения

**Funnel Chart и Bar Chart:**

| Элемент | Описание |
|---|---|
| Отображение | Side-by-side: две воронки рядом (основной период слева, сравнение справа) |
| Delta на этапе | Между воронками для каждого этапа отображается разница |
| Цвет delta | Зелёный (рост), красный (падение), серый (без изменений) |

**Delta-метрики на каждом этапе:**

| Метрика | Формат | Пример |
|---|---|---|
| Delta количества | +/- абсолютное число | «+12» или «-5» |
| Delta суммы | +/- абсолютное число | «+1 250 000 руб.» |
| Delta конверсии | +/- процентных пунктов | «+3.2 п.п.» или «-1.5 п.п.» |

**Таблица:**

Добавляются столбцы для периода сравнения и delta:

| Столбец | Описание |
|---|---|
| [метрика] (текущий) | Значение за основной период |
| [метрика] (сравнение) | Значение за период сравнения |
| Delta | Разница с цветовой индикацией |

## Сценарии

### Сценарий 1. Первый заход на отчёт (Best Guess)

1. Пользователь кликает «Отчёты» в главном меню Kaiten
2. В списке отчётов видит «Воронка продаж» и кликает на него
3. Открывается страница отчёта в контексте пространства «CRM»
4. Поскольку сохранённой конфигурации нет, запускается **алгоритм Best Guess**:
   - Система собирает все доски пространства: «Лиды» (3 колонки), «Сделки B2B» (5 колонок)
   - Колонки всех досок выстраиваются в единую последовательность (исключая `done`-колонки)
   - Этапы: Новый лид -> Квалификация -> КП отправлено -> Переговоры -> Договор -> Оплата
   - Won: «Закрыто (выиграно)» (последняя `done`-колонка)
   - Lost: «Закрыто (проиграно)» (остальные `done`-колонки)
   - На обеих досках обнаружено общее числовое поле «Бюджет сделки» -> режим `amount` доступен
5. **График отображается сразу** -- воронка за текущий месяц, режим «Количество» по умолчанию
6. Загрузка занимает <3 секунды
7. Пользователь видит трапецеидальную диаграмму с 6 этапами, конверсией между ними и количеством сделок
8. Над графиком нет алертов (confidence: high -- 2 done-колонки, одно общее поле суммы)
9. Пользователь переключает на «Сумма» -- воронка перестраивается по суммам сделок

### Сценарий 1а. Первый заход с алертом (неоднозначный Best Guess)

1. Пользователь открывает отчёт на пространстве «Продажи», где 3 доски
2. Best Guess обнаруживает 4 `done`-колонки (по 1-2 на каждой доске) и разные числовые поля на досках
3. **График строится сразу** по количеству сделок (fallback на `count`)
4. Над графиком показываются 2 алерта:
   - Warning: «Найдено 4 завершающих колонки. Последняя определена как "Выигран", остальные как "Проигран". Проверьте настройки.» [Настроить]
   - Warning: «На разных досках используются разные числовые поля. Воронка построена по количеству карточек.» [Настроить]
5. Пользователь видит воронку и может работать с ней
6. Пользователь кликает «Настроить» -- открывается боковая панель настроек

### Сценарий 2. Корректировка автоконфигурации

1. Пользователь находится на странице отчёта с автоконфигурацией (или ранее сохранённой)
2. Кликает иконку «Настройки» (шестерёнка) в верхнем правом углу
3. Открывается боковая панель с текущей конфигурацией:
   - Список досок пространства (с чекбоксами для включения/исключения)
   - Список этапов (колонки включённых досок) с чекбоксами и drag-and-drop
   - Выбор won/lost колонок
   - Поле суммы (выпадающий список числовых полей всех включённых досок)
4. Пользователь исключает доску «Письма» (нерелевантна для воронки)
5. Перетаскиванием меняет порядок двух этапов
6. Выбирает поле «Сумма контракта» как поле суммы
7. Нажимает «Сохранить» -- конфигурация сохраняется в БД (`auto_generated = false`), воронка перестраивается
8. Алерты Best Guess исчезают
9. При следующем открытии отчёта используется сохранённая конфигурация
10. Кнопка «Сбросить к автоопределению» внизу панели позволяет удалить сохранённую конфигурацию и вернуться к Best Guess

### Сценарий 3. Просмотр и анализ воронки

1. Пользователь открывает отчёт, загружается воронка с сохранёнными настройками

4. Между этапами — процент конверсии, цвет зависит от значения (зелёный >= 70%, оранжевый 40-69%, красный < 40%)
5. Замечает, что конверсия на этапе «КП отправлено -> Переговоры» красная (28%)
6. Наводит курсор на этап «КП отправлено» — tooltip показывает:
   - Количество: 64 сделки
   - Сумма: 8 320 000 руб.
   - Средний чек: 130 000 руб.
   - Среднее время: 6.3 дня
7. Пользователь переключается на вид «Таблица» для точных цифр

### Сценарий 4. Фильтрация

## Данные

### Входные данные для построения воронки

```typescript
interface FunnelConfig {
  spaceId: number;                    // ID пространства
  boardIds: number[];                 // ID досок, входящих в воронку
  stages: FunnelStage[];              // Этапы воронки (упорядоченные)
  winColumnIds: number[];             // ID колонок "Выигран"
  lossColumnIds: number[];            // ID колонок "Проигран"
  amountFieldId: number | null;       // ID кастомного поля суммы (null если не определено)
  amountFieldMapping: Record<number, number> | null; // Маппинг board_id -> field_id (если разные поля на досках)
  metricMode: 'count' | 'amount';    // Режим метрики (по умолчанию 'count')
  autoGenerated: boolean;             // true если конфигурация сгенерирована Best Guess (не сохранена в БД)
}

interface FunnelStage {
  columnId: number;                   // ID колонки
  boardId: number;                    // ID доски, которой принадлежит колонка
  name: string;                       // Название колонки
  displayName: string;                // Название для отображения (с суффиксом доски при дубликатах)
  order: number;                      // Глобальный порядок в воронке (0, 1, 2, ...)
  included: boolean;                  // Включён ли этап в воронку
}
```

### Фильтры

```typescript
interface FunnelFilters {
  period: PeriodFilter;
  ownerIds: number[];                 // ID ответственных (пустой массив = все)
  tagIds: number[];                   // ID тегов (пустой массив = все)
  amountRange: {
    min: number | null;               // Минимальная сумма (null = без ограничения)
    max: number | null;               // Максимальная сумма (null = без ограничения)
  };
}

interface PeriodFilter {
  type: 'today' | 'last_7_days' | 'current_month' | 'previous_month'
        | 'current_quarter' | 'current_year' | 'custom';
  startDate: string;                  // ISO date (YYYY-MM-DD)
  endDate: string;                    // ISO date (YYYY-MM-DD)
}
```

## Валидация

### FE-валидация

| Поле | Правило | Поведение при ошибке |
|---|---|---|
| Пространство | Определено в контексте навигации | Если пространство не определено -- показывается сообщение «Выберите пространство для построения воронки» |
| Доски (настройки) | Минимум 1 включённая доска | Кнопка «Сохранить» заблокирована; текст «Включите минимум 1 доску» |
| Этапы (настройки) | Минимум 2 включённых | Кнопка «Сохранить» заблокирована; текст «Выберите минимум 2 этапа» |
| Период -- начальная дата | Не позже конечной | Подсветка красным, tooltip «Начальная дата не может быть позже конечной» |
| Сумма от | >= 0 | Не принимается отрицательное значение |
| Сумма до | >= значения «от» | Подсветка красным при значении меньше «от» |
| Сумма от/до | Числовое значение | Нечисловые символы не вводятся (input type=number) |
| Сумма от/до | Доступность | Поля задизейблены, если `metricMode = 'count'` и поле суммы не определено |

### BE-валидация

| Поле | Правило | HTTP-код | Сообщение |
|---|---|---|---|
| spaceId | Пространство существует и пользователь имеет доступ | 403 | «Нет доступа к пространству» |
| spaceId | В пространстве есть хотя бы 1 неархивная доска | 400 | «В пространстве нет досок» |
| boardIds | Все доски принадлежат указанному пространству | 400 | «Доска не принадлежит пространству» |
| stages | Минимум 2 валидных columnId, принадлежащих указанным boardIds | 400 | «Необходимо минимум 2 этапа» |
| period.startDate | Валидная дата, не в будущем | 400 | «Некорректная начальная дата» |
| period.endDate | >= startDate | 400 | «Конечная дата не может быть раньше начальной» |
| amountFieldId | Поле существует, является числовым и принадлежит одной из досок | 400 | «Поле суммы не найдено или имеет неверный тип» |
| ownerIds | Все ID -- валидные участники досок пространства | 400 | «Указан несуществующий ответственный» |
| metricMode | Одно из: `'count'`, `'amount'` | 400 | «Некорректный режим метрики» |

## Продуктовые метрики

### Зачем собираем метрики

Отчёт «Воронка продаж» — один из ключевых инструментов для CRM-пользователей Kaiten. Метрики помогут понять:

- **Нашли ли пользователи отчёт?** — сколько людей из CRM-пользователей доходят до воронки
- **Могут ли настроить?** — есть ли проблемы с онбордингом (первая настройка)
- **Получают ли ценность?** — используют ли аналитические возможности (фильтры, drill-down, сравнение)
- **Возвращаются ли?** — становится ли отчёт привычным инструментом
- **Что используют чаще всего?** — приоритизация доработок

### 1. Воронка использования: открытие -> Best Guess -> анализ -> действие

Цель: понять, на каком шаге пользователи «отваливаются» и где есть проблемы с UX. С переходом на Best Guess ключевой момент -- пользователь видит график сразу, без настройки. Метрики фокусируются на качестве автоконфигурации и конверсии в аналитику.

| Событие | Где срабатывает | Параметры | Зачем |
|---|---|---|---|
| `funnel_report_opened` | Переход на страницу отчёта | `user_id`, `company_id`, `space_id`, `has_saved_config` (true/false), `config_type` (`auto`/`saved`) | Верх воронки -- сколько людей открывают отчёт; какая доля работает на Best Guess vs. сохранённом конфиге |
| `funnel_auto_config_applied` | Best Guess определил конфигурацию | `user_id`, `space_id`, `boards_count`, `stages_count`, `confidence` (`high`/`medium`/`low`), `metric_mode` (`count`/`amount`), `alerts_count`, `alert_codes[]` | Качество автоконфигурации: какая доля high confidence; как часто срабатывает fallback на count |
| `funnel_best_guess_alert_shown` | Показ алерта Best Guess над графиком | `user_id`, `space_id`, `alert_code`, `alert_type` (`info`/`warning`/`error`) | Как часто пользователи видят алерты; какие типы неоднозначностей чаще |
| `funnel_best_guess_alert_action` | Клик на действие в алерте (Настроить / Перейти) | `user_id`, `alert_code`, `action` (`configure`/`dismiss`/`navigate`) | Реагируют ли на алерты; что предпочитают -- настроить или закрыть |
| `funnel_config_saved` | Сохранение пользовательской конфигурации | `user_id`, `space_id`, `boards_count`, `stages_count`, `has_amount_field`, `excluded_stages_count`, `was_auto_generated`, `time_to_configure` | Сколько переходят от Best Guess к ручной настройке; сколько времени тратят |
| `funnel_config_reset` | Сброс к автоопределению | `user_id`, `space_id` | Возвращаются ли к Best Guess после ручной настройки |
| `funnel_metric_mode_switched` | Переключение «Количество / Сумма» | `user_id`, `from_mode`, `to_mode` | Как часто переключают; предпочитают count или amount |
| `funnel_filter_applied` | Применение любого фильтра | `user_id`, `filter_type` (period/owner/tags/amount), `filter_value_count` | Какие фильтры популярны |
| `funnel_drilldown_opened` | Клик по этапу воронки (drill-down) | `user_id`, `stage_name`, `deal_count` | Сколько используют drill-down |
| `funnel_card_opened` | Переход в карточку из drill-down | `user_id`, `card_id`, `stage_name` | Доходят ли до конкретных действий |
| `funnel_comparison_enabled` | Включение режима сравнения | `user_id`, `base_period`, `comparison_period` | Используют ли сравнение |
| `funnel_exported` | Клик на экспорт | `user_id`, `export_format` (csv/xlsx/png), `has_comparison` | Формат и контекст экспорта |
| `funnel_view_switched` | Переключение вида визуализации | `user_id`, `view_type` (funnel/bar/table), `previous_view` | Какой вид предпочитают |

**Ключевые метрики воронки использования:**

| Метрика | Формула | Целевое значение | Зачем |
|---|---|---|---|
| **Time to first chart** | Время от `funnel_report_opened` до отображения графика | < 3 сек | Основная метрика Best Guess -- пользователь видит результат мгновенно |
| **Best Guess acceptance rate** | Пользователи, НЕ открывшие настройки после `funnel_auto_config_applied` / все с auto_config | > 60% | Если ниже -- автоконфигурация недостаточно точная |
| **High confidence rate** | `funnel_auto_config_applied` с `confidence=high` / все auto_config | > 50% | Качество алгоритма Best Guess |
| **Alert action rate** | `funnel_best_guess_alert_action` с `action=configure` / все `funnel_best_guess_alert_shown` | 20-40% | Если >50% -- алерты провоцируют действие (хорошо). Если <10% -- алерты игнорируются |
| **Config save rate** | Уникальные user_id с `funnel_config_saved` / все user_id с `funnel_report_opened` | 15-30% | Если >50% -- Best Guess работает плохо. Если <5% -- настройки не нужны |
| **Filter usage rate** | Уникальные user_id с `funnel_filter_applied` / уникальные user_id с `funnel_report_opened` | > 50% | Если ниже -- фильтры не нужны или не заметны |
| **Drill-down rate** | Уникальные user_id с `funnel_drilldown_opened` / уникальные user_id с `funnel_report_opened` | > 30% | Если ниже -- drill-down не очевиден |
| **Action rate** | Уникальные user_id с `funnel_card_opened` / уникальные user_id с `funnel_drilldown_opened` | > 40% | Если ниже -- drill-down не ведёт к действиям |
| **Comparison usage rate** | Уникальные user_id с `funnel_comparison_enabled` / уникальные user_id с `funnel_report_opened` | > 15% | Оценка востребованности функции |
| **Export rate** | Уникальные user_id с `funnel_exported` / уникальные user_id с `funnel_report_opened` | > 10% | Если ниже -- не экспортируют или нет потребности |

### 2. Возвращаемость и привычка

Цель: понять, становится ли отчёт регулярным инструментом работы.

| Метрика | Формула | Целевое значение | Зачем |
|---|---|---|---|
| **Weekly active users** | Уникальные user_id с `funnel_report_opened` за неделю | Рост +5% в неделю (первые 3 месяца) | Растёт ли аудитория |
| **Retention D7** | Пользователи, открывшие отчёт через 7 дней после первого визита / все, кто открыл впервые | > 30% | Возвращаются ли через неделю |
| **Retention D30** | Пользователи, открывшие отчёт через 30 дней / все, кто открыл впервые | > 20% | Становится ли привычкой |
| **Sessions per user per week** | Среднее кол-во `funnel_report_opened` на пользователя в неделю | 2-4 | Частота использования |
| **Power users share** | Пользователи с >= 5 сессиями в неделю / все пользователи | 10-20% | Есть ли ядро активных пользователей |

### 3. Популярность фильтров

Цель: понять, какие фильтры нужны, а какие — лишние.

Считаем на основе `funnel_filter_applied`:

| Метрика | Формула | Зачем |
|---|---|---|
| **Filter type distribution** | Группировка `funnel_filter_applied` по `filter_type` | Какие фильтры используют чаще |
| **Period preset distribution** | Группировка по `filter_value` при `filter_type=period` | Какие предустановленные периоды популярны |
| **Avg filters per session** | Среднее количество уникальных `filter_type` за сессию | Сколько фильтров применяют одновременно (если > 3 — нужны пресеты) |
| **Custom period rate** | `filter_type=period AND value=custom` / все `filter_type=period` | Достаточно ли предустановленных периодов |
| **Owner filter rate** | Сессии с `filter_type=owner` / все сессии | Как часто фильтруют по менеджеру |

### 4. Использование визуализаций

Цель: определить, какой вид отчёта наиболее востребован.

| Метрика | Формула | Зачем |
|---|---|---|
| **Default view distribution** | Последний сохранённый вид по user_id | Какой вид выбирают как основной |
| **View switch rate** | Сессии с `funnel_view_switched` / все сессии | Как часто переключают вид |
| **Funnel chart preference** | Пользователи с `view_type=funnel` как последний сохранённый / все | Долю предпочтителей каждого вида |

### 5. Эффективность drill-down

Цель: понять, даёт ли drill-down реальную ценность.

| Метрика | Формула | Зачем |
|---|---|---|
| **Drill-down to action rate** | `funnel_card_opened` / `funnel_drilldown_opened` | Конвертируется ли drill-down в действия |
| **Avg cards viewed per drilldown** | Среднее `funnel_card_opened` за один `funnel_drilldown_opened` | Сколько карточек просматривают |
| **Drilldown sort usage** | Сессии с изменением сортировки / все drill-down сессии | Нужна ли сортировка |
| **Drilldown export rate** | Экспорт из drill-down / все drill-down сессии | Нужен ли экспорт списка |
| **Most drilled-down stages** | Группировка `funnel_drilldown_opened` по `stage_name` | Какие этапы вызывают больше вопросов |

### Сводная карта событий по элементам интерфейса

```
Главное меню
  └─ «Отчёты» -> «Воронка продаж» -> funnel_report_opened

  ├─ Панель фильтров

  │   ├─ Фильтр ответственного -> funnel_filter_applied (filter_type=owner)
  │   ├─ Фильтр тегов -> funnel_filter_applied (filter_type=tags)
  │   └─ Фильтр суммы -> funnel_filter_applied (filter_type=amount)
  │
  ├─ Визуализация
  │   ├─ Переключатель видов -> funnel_view_switched
  │   ├─ Переключатель «Кол-во / Сумма» -> funnel_metric_mode_switched
  │   ├─ Клик по этапу -> funnel_drilldown_opened
  │   └─ Кнопка «Сравнить периоды» -> funnel_comparison_enabled
  │
  ├─ Drill-down панель
  │   ├─ Клик на карточку -> funnel_card_opened
  │   └─ Кнопка «Экспорт списка» -> funnel_exported (format=csv, source=drilldown)
  │
  ├─ Настройки (шестерёнка)
  │   ├─ Кнопка «Сохранить» -> funnel_config_saved
  │   └─ Кнопка «Сбросить к автоопределению» -> funnel_config_reset

  └─ Кнопка «Экспорт»
```

## Что не входит в первую версию

| Функциональность | Причина исключения | Кандидат для версии |
|---|---|---|
| Sankey-диаграмма перетоков | Высокая сложность реализации, нишевая потребность | v2-v3 |
| AI-рекомендации по оптимизации воронки | Требует ML-модели и накопления данных | v3+ |
| Прогнозирование выручки | Требует статистической модели и исторических данных | v2 |
| Уведомления при падении конверсии | Требует системы алертов и пороговых значений | v2 |
| Вложенные воронки (под-этапы) | Не поддерживается текущей моделью данных Kaiten | v3+ |
| Фильтрация по кастомным полям (кроме суммы) | Требует универсального механизма фильтров | v2 |
| Когортный анализ | Высокая сложность, нишевая потребность | v2-v3 |
| Публичные ссылки на отчёт | Требует системы шаринга | v2 |
| Шаблоны отчётов (сохранённые комбинации фильтров) | Усложняет MVP | v2 |
| Real-time обновление (WebSocket) | Избыточно для отчётов, которые открывают 1-5 раз в день | v2+ |
