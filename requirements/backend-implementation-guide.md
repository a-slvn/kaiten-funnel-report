## Что мы строим

Отчёт «Воронка продаж» показывает, как сделки (карточки на досках Kaiten) проходят через этапы воронки (колонки досок пространства). Фронтенд уже есть — горизонтальный bar chart с конверсиями между этапами, KPI-карточки, таблица, drill-down.

**Ключевое архитектурное решение (v2.0):** Воронка строится на уровне **пространства** (`space_id`), а не отдельной доски. Одно пространство может содержать несколько досок, колонки которых объединяются в единую последовательность этапов. При первом открытии отчёта график **строится сразу** на основе алгоритма Best Guess — без обязательной предварительной настройки.

Бекенд должен:
1. **Определить конфигурацию воронки** — загрузить сохранённую или автосгенерировать через Best Guess
2. Загрузить доски пространства и построить единую последовательность этапов
3. Обработать историю перемещений карточек **всех досок** пространства
4. Определить, какие сделки попадают в отчёт за выбранный период
5. Посчитать метрики по каждому этапу и по всей воронке
6. Отдать результат через API

## Самое важное: когортный подход

Это ключевая идея, без которой всё остальное не имеет смысла.

### Как определяется, какие сделки попадают в отчёт

Каждая сделка привязывается к периоду по **дате первого входа на любой этап воронки**. Это называется «дата-якорь». Например, если сделка впервые оказалась на этапе «Квалификация» 20 мая — её дата-якорь = 20 мая, и она навсегда принадлежит майской когорте.

**Когорта периода** — это все сделки, чья дата-якорь попадает в выбранный диапазон дат.

### Пример

Сделка «РЖД» попала на «Квалификацию» 20 мая, затем прошла через «Встречу» 25 мая, «Предложение» 3 июня и была выиграна 28 июня.

- **Отчёт за май** — сделка **есть** в отчёте. Показываются все этапы, включая те, что произошли уже в июне (Предложение, Выигран).
- **Отчёт за июнь** — сделки **нет** в отчёте. Хотя она активно двигалась в июне, её первый вход был в мае.

### Почему именно так

- Каждая сделка попадает **ровно в один** период — нет двойного счёта.
- Числа на этапах воронки убывают сверху вниз — каждый следующий этап не больше предыдущего.
- Drill-down по любому числу показывает конкретные карточки.
- Когорта «дозревает» со временем — конверсия растёт по мере того, как сделки закрываются.

### Что значит «дозревание» когорты

Если открыть отчёт за май в июне — сделка РЖД ещё в процессе (не выиграна). Если открыть тот же отчёт за май в июле — сделка уже выиграна, конверсия майской когорты выросла.

Это правильное поведение: победа учитывается **независимо от даты** — важно лишь то, что сделка из этой когорты в итоге победила.

## Два типа метрик

В отчёте есть два принципиально разных типа метрик.

**Когортные метрики** считаются по когорте выбранного периода. Сюда входят: количество сделок на этапе, суммы, конверсии, время на этапе, цикл сделки, скорость воронки, потерянные сделки, коэффициент потерь. Эти метрики зависят от ширины периода — месяц покажет больше сделок, чем неделя.

**Snapshot-метрики** показывают состояние воронки на конец периода, независимо от когорты. Сюда входят: объём воронки (pipeline), взвешенный объём, зависшие сделки. Эти метрики одинаковы для любого периода с одинаковой конечной датой.

### Два режима метрик: «Сумма» и «Количество»

Воронка поддерживает два режима отображения (`metric_mode`):

**Режим `amount`** — основная метрика по этапам = сумма сделок. Доступен, только если определено числовое поле суммы (через автоконфигурацию или настройку).

**Режим `count`** — основная метрика по этапам = количество карточек. Используется по умолчанию, а также как fallback, если поле суммы не определено.

| Метрика | `amount` | `count` |
|---------|----------|---------|
| Значение этапа (основная цифра) | SUM(deal_amount) | COUNT(DISTINCT card_id) |
| Средний чек | SUM / COUNT(с суммой) | Не отображается |
| Pipeline value | SUM(deal_amount) | COUNT(DISTINCT card_id) на этапах |
| Weighted pipeline | SUM(deal_amount * probability) | Не отображается |
| Velocity | deals * win_rate * avg_size / cycle | Не отображается |
| Конверсия | Без изменений (всегда по количеству) | Без изменений |
| Время на этапе | Без изменений | Без изменений |

## Алгоритм расчёта, шаг за шагом

### Шаг 0. Автоконфигурация Best Guess

Этот шаг выполняется **перед** загрузкой конфигурации, если для пространства нет сохранённого конфига.

#### Когда вызывается автоконфигурация

1. Пользователь открывает отчёт для пространства, у которого нет сохранённой записи `funnel_config`.
2. Пользователь удалил ранее сохранённый конфиг (DELETE /config/:id).
3. Явный вызов через endpoint `GET /api/v1/reports/funnel/auto-config?space_id=X`.

#### Как работает алгоритм

Алгоритм принимает `space_id` и возвращает готовую конфигурацию: последовательность этапов, колонки won/lost, режим метрики, алерты для пользователя. Полное описание — в `research/best-guess-algorithm.md`.

**Шаг 0.1. Сбор досок пространства.**
Получить все неархивные доски, отсортированные сверху вниз, слева направо (`row_sort_order ASC, sort_order ASC`). Если досок нет — ошибка `NO_BOARDS`, график не строится.

**Шаг 0.2. Сбор колонок всех досок.**
Для каждой доски получить колонки, отсортированные по `sort_order` слева направо.

**Шаг 0.3. Построение единой последовательности этапов.**
Колонки всех досок объединяются последовательно: доска 1 (все не-done колонки) -> доска 2 -> ... -> доска N. Каждая колонка получает глобальный `sort_order`. Колонки с `column_type = 'done'` исключаются из этапов (они — кандидаты для won/lost). Если не осталось ни одного этапа — ошибка `ALL_COLUMNS_DONE`.

**Шаг 0.4. Определение won/lost колонок.**
На основе done-колонок:

| Done-колонок | Won | Lost | Уверенность |
|-------------|-----|------|-------------|
| 0 | Последний этап (убирается из stages) | Пусто | `low` |
| 1 | Эта done-колонка | Пусто | `medium` |
| 2 | Последняя done по глобальному порядку | Первая done | `high` |
| 3+ | Последняя done | Все остальные done | `medium` |

**Шаг 0.5. Определение поля суммы сделки.**
Поиск числовых кастомных полей на всех досках. Если одно общее поле по имени на всех досках — `metric_mode = 'amount'`. Если общего поля нет — `metric_mode = 'count'` (fallback на количество карточек). Подробная стратегия — в `research/best-guess-algorithm.md`, раздел «Шаг 5».

**Шаг 0.6. Сборка результата.**
Формируется `AutoFunnelConfig` с флагом `auto_generated = true`. Конфигурация НЕ сохраняется в базу — вычисляется на лету при каждом открытии отчёта без сохранённого конфига.

#### Псевдокод сервиса автоконфигурации

```python
def get_funnel_config(space_id, company_id):
    """
    Главная точка входа: загрузка или генерация конфигурации воронки.
    Вызывается при каждом запросе к отчёту.
    """
    # 1. Проверить наличие сохранённого конфига
    saved_config = db.find_one(
        "SELECT * FROM funnel_config WHERE space_id = :space_id AND company_id = :company_id",
        space_id=space_id, company_id=company_id
    )

    if saved_config is not None:
        # Сохранённый конфиг имеет приоритет
        return saved_config, alerts=[]

    # 2. Нет сохранённого конфига — запускаем Best Guess
    return best_guess_auto_config(space_id, company_id)


def best_guess_auto_config(space_id, company_id):
    """
    Алгоритм автоконфигурации. Полная реализация — в best-guess-algorithm.md.
    """
    # Шаг 0.1: Доски
    boards = get_space_boards(space_id, company_id, archived=False)
    if not boards:
        raise AutoConfigError('NO_BOARDS')

    board_ids = [b.id for b in boards]

    # Шаг 0.2: Колонки
    columns_by_board = get_columns_by_board(board_ids)

    # Шаг 0.3: Этапы + done-колонки
    stages, done_columns = build_unified_stages(boards, columns_by_board)
    if not stages:
        raise AutoConfigError('ALL_COLUMNS_DONE')

    # Шаг 0.4: Won/Lost
    won_lost = determine_won_lost(done_columns, stages)

    # Шаг 0.5: Поле суммы
    amount = determine_amount_field(board_ids)

    # Шаг 0.6: Сборка
    return AutoFunnelConfig(
        space_id=space_id,
        board_ids=board_ids,
        stages=stages,
        win_column_ids=won_lost['win'],
        loss_column_ids=won_lost['loss'],
        deal_amount_field_id=amount['field_id'],
        metric_mode=amount['metric_mode'],
        auto_generated=True
    ), alerts=won_lost['alerts'] + amount['alerts']
```

#### Алерты Best Guess

При неоднозначной автоконфигурации бекенд возвращает алерты в ответе API. Алерты не блокируют отображение графика (кроме типа `error`). Ключевые алерты:

| Код | Тип | Когда |
|-----|-----|-------|
| `NO_BOARDS` | error | В пространстве нет досок |
| `ALL_COLUMNS_DONE` | error | Все колонки = done, нет этапов |
| `NO_DONE_COLUMNS` | warning | Нет done-колонок, последний этап использован как won |
| `SINGLE_DONE_COLUMN` | info | Одна done-колонка = won, loss не определён |
| `MULTIPLE_DONE_COLUMNS` | warning | 3+ done-колонок, назначение может быть неточным |
| `NO_AMOUNT_FIELD` | info | Нет числовых полей, воронка по количеству |
| `DIFFERENT_AMOUNT_FIELDS` | warning | Разные поля суммы на досках, fallback на count |

Полный каталог алертов — в `research/best-guess-algorithm.md`, раздел «Условия показа алертов».

### Конкурентное создание конфигурации (race condition handling)

При сохранении конфигурации воронки (POST /config) возможны два типа конкурентных проблем.

#### Проблема 1: Два параллельных запроса на создание конфига

**Сценарий:** Два пользователя одного пространства одновременно открывают отчёт, оба видят auto-config, оба нажимают «Сохранить». Оба запроса проходят валидацию, но unique constraint `(space_id, company_id)` допускает только одну запись.

**Решение: upsert-like поведение.**

Вместо классической схемы «проверить — вставить» (которая подвержена race condition) используется атомарная операция на уровне БД:

```sql
-- Шаг 1: Атомарный INSERT с обработкой конфликта
INSERT INTO funnel_config (space_id, company_id, name, metric_mode, ...)
VALUES (:space_id, :company_id, :name, :metric_mode, ...)
ON CONFLICT (space_id, company_id) DO NOTHING
RETURNING id;

-- Шаг 2: Если INSERT не вернул id (конфиг уже существует) — загрузить существующий
SELECT * FROM funnel_config
WHERE space_id = :space_id AND company_id = :company_id;
```

**Логика сервиса:**

1. Выполнить валидацию входных данных (stages, columns, fields)
2. Выполнить `INSERT ... ON CONFLICT DO NOTHING RETURNING id`
3. Если `RETURNING` вернул `id` — конфиг создан, вернуть `201 Created` с `already_existed = false`
4. Если `RETURNING` не вернул ничего — конфиг уже существует:
   - Загрузить существующий конфиг по `(space_id, company_id)`
   - Вернуть `200 OK` с `already_existed = true`

**Почему не `SELECT FOR UPDATE` + check + insert:**
- `SELECT FOR UPDATE` требует явной транзакции и блокирует запись до конца транзакции
- Если записи ещё нет, `SELECT FOR UPDATE` ничего не заблокирует — race condition остаётся
- `INSERT ON CONFLICT` — атомарная операция, обрабатываемая движком БД, без окна для гонки

**Почему не `409 CONFIG_EXISTS`:**
- С точки зрения пользователя, результат один: конфиг для пространства существует
- Показывать ошибку второму пользователю некорректно — он ничего неправильного не сделал
- Upsert-поведение проще для клиента: всегда получаешь валидный конфиг, различай по HTTP-коду

#### Проблема 2: Структура пространства изменилась после генерации auto-config

**Сценарий:** Пользователь открыл отчёт, получил auto-config. Пока он настраивал параметры, администратор добавил новую колонку на доску (или удалил доску, или переименовал кастомное поле). Пользователь нажимает «Сохранить» — и сохраняет устаревшую конфигурацию.

**Решение: optimistic locking через `structure_hash`.**

**Что такое `structure_hash`:**
Хеш вычисляется как SHA-256 от детерминированной строки, включающей:
- Отсортированные `board_id` всех неархивных досок пространства
- Отсортированные `column_id` всех колонок этих досок
- Отсортированные `custom_field_id` числовых кастомных полей этих досок

```python
def compute_structure_hash(space_id, company_id):
    boards = get_space_boards(space_id, company_id, archived=False)
    board_ids = sorted([b.id for b in boards])

    column_ids = sorted([
        col.id
        for b in boards
        for col in get_board_columns(b.id)
    ])

    field_ids = sorted([
        f.id
        for b in boards
        for f in get_board_custom_fields(b.id)
        if f.field_type == 'number'
    ])

    payload = f"{board_ids}|{column_ids}|{field_ids}"
    return hashlib.sha256(payload.encode()).hexdigest()
```

**Логика проверки при POST /config:**

1. Клиент передаёт `expected_structure_hash` (полученный ранее из `GET /auto-config`)
2. Бекенд вычисляет текущий `structure_hash` пространства
3. Сравнение:
   - Хеши совпадают — структура не изменилась, продолжить сохранение
   - Хеши не совпадают — вернуть `409 STRUCTURE_CHANGED` с телом:
     - Код ошибки `STRUCTURE_CHANGED`
     - `expected_hash` и `current_hash` для отладки
     - Свежий `AutoConfigResponse` (чтобы клиент мог показать обновлённую конфигурацию)
   - `expected_structure_hash` не передан — пропустить проверку (backward-compatible)

**Порядок проверок при POST /config:**

```
1. Валидация входных данных (400 ошибки)
2. Проверка доступа (403)
3. Проверка structure_hash (409 STRUCTURE_CHANGED) — если передан
4. INSERT ON CONFLICT (201 или 200)
```

Проверка `structure_hash` выполняется **до** попытки вставки. Если структура изменилась — нет смысла пытаться сохранить устаревший конфиг.

**Когда `expected_structure_hash` не передаётся:**
- При ручной настройке конфига (пользователь сам выбирает колонки в UI настроек)
- При программном создании через API (интеграции)
- В этих случаях проверка пропускается — пользователь/интеграция берёт ответственность за актуальность данных

### Шаг 1. Загрузить конфигурацию воронки

Из конфигурации воронки берём: какое пространство (`space_id`), какие доски входят (`board_ids`), какие колонки являются этапами воронки и в каком порядке (с указанием `board_id` для каждой колонки), какие колонки означают победу, какие — проигрыш, режим метрики (`metric_mode`), и какое кастомное поле (или маппинг полей) хранит сумму сделки.

**Логика определения конфигурации:**

1. Если в запросе передан `funnel_config_id` — загрузить этот конфиг из БД.
2. Если передан `space_id` (без `funnel_config_id`) — искать сохранённый конфиг по `(space_id, company_id)`. Если найден — использовать его. Если не найден — **запустить автоконфигурацию Best Guess** (Шаг 0).
3. Если не передан ни `funnel_config_id`, ни `space_id` — вернуть ошибку.

**Проверка `auto_generated`:**
Если конфиг автоматически сгенерирован (`auto_generated = true`), в ответе API передаётся флаг `auto_generated` и массив алертов. Фронтенд показывает алерты пользователю.

**Загрузка досок пространства:**
После получения конфига загрузить актуальный список досок по `board_ids`. Для каждой доски проверить, что она всё ещё существует и принадлежит пространству. Архивные доски, попавшие в `board_ids` после архивации, помечаются предупреждением.

**Валидация `deal_amount_field_mapping`:**
После загрузки конфигурации (сохранённой или автосгенерированной) необходимо проверить корректность всех полей суммы, указанных в маппинге. Поле может стать невалидным, если его удалили, изменили тип или перенесли на другую доску после сохранения конфига.

Шаги валидации:

1. **Проверка существования.** Для каждой записи `(board_id, field_id)` из `deal_amount_field_mapping` проверить, что `field_id` существует в таблице `custom_fields`.
2. **Проверка типа.** Для каждого найденного поля проверить, что `custom_fields.field_type = 'number'`. Поле, тип которого изменён на нечисловой (например, `text`, `select`), невалидно.
3. **Проверка принадлежности доске.** Для каждой записи `(board_id, field_id)` проверить, что `custom_fields.board_id = mapping_key` (то есть поле принадлежит той доске, для которой указано в маппинге). Поле, перенесённое на другую доску, невалидно.
4. **Валидация глобального поля `deal_amount_field_id`.** Если задан `deal_amount_field_id` (общее поле, не через маппинг), к нему применяются те же проверки: существование и `field_type = 'number'`. Проверка принадлежности доске не требуется (общее поле может быть на любой доске).

Обработка невалидных записей:

- **Невалидные записи в маппинге** исключаются из `deal_amount_field_mapping`. Для каждой исключённой записи формируется warning alert `INVALID_FIELD_MAPPING` с указанием `board_id`, `field_id` и причины (`field_not_found`, `wrong_field_type`, `wrong_board`).
- **Если после исключения маппинг непуст** — работа продолжается с очищенным маппингом. Для досок с исключёнными записями fallback на `deal_amount_field_id` (если задан и валиден).
- **Если после исключения маппинг пуст И `deal_amount_field_id` тоже невалиден (или не задан)** — автоматический fallback на `metric_mode = 'count'`. Формируется алерт `FALLBACK_TO_COUNT` с пояснением причины.
- **Если `deal_amount_field_id` невалиден, но маппинг непуст** — `deal_amount_field_id` обнуляется, маппинг используется как основной источник.

Псевдокод валидации:

```python
def validate_amount_field_mapping(config, board_ids):
    """
    Валидация deal_amount_field_mapping после загрузки конфига.
    Вызывается ПЕРЕД началом расчётов.
    """
    alerts = []
    mapping = config.deal_amount_field_mapping or {}
    valid_mapping = {}

    if mapping:
        # Одним запросом получить все поля из маппинга
        field_ids = list(mapping.values())
        existing_fields = db.query(
            """SELECT id, field_type, board_id
               FROM custom_fields
               WHERE id = ANY(:field_ids)""",
            field_ids=field_ids
        )
        fields_by_id = {f.id: f for f in existing_fields}

        for board_id_key, field_id in mapping.items():
            field = fields_by_id.get(field_id)

            if field is None:
                alerts.append(Alert(
                    code='INVALID_FIELD_MAPPING',
                    type='warning',
                    details={
                        'board_id': board_id_key,
                        'field_id': field_id,
                        'reason': 'field_not_found'
                    }
                ))
            elif field.field_type != 'number':
                alerts.append(Alert(
                    code='INVALID_FIELD_MAPPING',
                    type='warning',
                    details={
                        'board_id': board_id_key,
                        'field_id': field_id,
                        'reason': 'wrong_field_type',
                        'actual_type': field.field_type
                    }
                ))
            elif str(field.board_id) != str(board_id_key):
                alerts.append(Alert(
                    code='INVALID_FIELD_MAPPING',
                    type='warning',
                    details={
                        'board_id': board_id_key,
                        'field_id': field_id,
                        'reason': 'wrong_board',
                        'actual_board_id': field.board_id
                    }
                ))
            else:
                valid_mapping[board_id_key] = field_id

        config.deal_amount_field_mapping = valid_mapping

    # Валидация глобального deal_amount_field_id
    if config.deal_amount_field_id:
        global_field = db.find_one(
            """SELECT id, field_type
               FROM custom_fields
               WHERE id = :field_id""",
            field_id=config.deal_amount_field_id
        )
        if global_field is None or global_field.field_type != 'number':
            alerts.append(Alert(
                code='INVALID_FIELD_MAPPING',
                type='warning',
                details={
                    'field_id': config.deal_amount_field_id,
                    'reason': 'field_not_found' if global_field is None
                              else 'wrong_field_type'
                }
            ))
            config.deal_amount_field_id = None

    # Fallback на count, если не осталось валидных полей суммы
    if not config.deal_amount_field_mapping and not config.deal_amount_field_id:
        if config.metric_mode == 'amount':
            config.metric_mode = 'count'
            alerts.append(Alert(
                code='FALLBACK_TO_COUNT',
                type='warning',
                details={
                    'reason': 'no_valid_amount_fields',
                    'message': 'Все поля суммы невалидны. '
                               'Воронка переключена в режим «Количество».'
                }
            ))

    return alerts
```

Алерт `INVALID_FIELD_MAPPING`:

| Код | Тип | Когда | Действие |
|-----|-----|-------|----------|
| `INVALID_FIELD_MAPPING` | warning | Поле из маппинга удалено, изменён тип или перенесено на другую доску | Невалидная запись исключается из маппинга |
| `FALLBACK_TO_COUNT` | warning | Все поля суммы невалидны (и маппинг, и глобальное поле) | Автоматический fallback на `metric_mode = 'count'` |

### Шаг 2. Отфильтровать карточки

Применить фильтры пользователя к карточкам **всех досок пространства** (`board_ids`): по ответственному, по источнику сделки, по тегам, по диапазону суммы, по архивности. Результат — набор карточек, которые проходят фильтры. Все дальнейшие расчёты идут только по этим карточкам.

**Мультидосочная фильтрация:**

```
WHERE c.board_id = ANY(:board_ids)        -- все доски пространства
  AND (c.archived = false OR :include_archived = true)
  AND (:responsible_ids IS NULL OR c.responsible_id = ANY(:responsible_ids))
  ...
```

**Фильтрация по сумме с учётом маппинга полей:**
Если на разных досках сумма хранится в разных кастомных полях (настроено через `deal_amount_field_mapping`), фильтр по `amount_min` / `amount_max` должен использовать поле конкретной доски карточки.

```
-- Для каждой карточки определяем field_id суммы:
-- приоритет: deal_amount_field_mapping[card.board_id] > deal_amount_field_id > null
COALESCE(
    (:deal_amount_field_mapping ->> c.board_id::text)::int,
    :deal_amount_field_id
)
```

Подробный SQL — в `data-model/calculation-spec.md`, раздел 3.

### Шаг 3. Построить историю прохождения сделок через воронку

Это ключевой и самый трудоёмкий шаг. Нужно взять все перемещения карточек **всех досок пространства** и превратить их в структурированную историю пребывания каждой сделки на каждом этапе.

**Источник данных:** `card_movements` для всех карточек из `board_ids`:

```
WHERE c.board_id = ANY(:board_ids)        -- все доски воронки
```

Для каждого перемещения карточки смотрим:
- Если карточка **пришла** на колонку, которая является этапом воронки — записываем вход на этап (фиксируем время входа).
- Если карточка **ушла** с колонки, которая является этапом воронки — закрываем запись пребывания (фиксируем время выхода, считаем длительность, запоминаем, куда ушла дальше).
- Если карточка вернулась на этап, где уже была — создаём новую запись с номером визита 2 (или 3, если это третий раз).

**Мультидосочные особенности (v2.0):**

1. **`board_id` в каждой записи.** Каждая запись `funnel_deal_history` содержит `board_id` карточки (из `cards.board_id`). Это критично для маппинга поля суммы и кросс-board аналитики.

2. **Глобальный `stage_sort_order`.** Порядок этапов — сквозной по всем доскам. Например, если доска «Лиды» содержит этапы 1-3, а доска «Сделки» — этапы 4-6, то `sort_order` идёт от 1 до 6 непрерывно. Этот порядок определяет, что значит «прошла дальше» и «вернулась назад».

3. **Маппинг `deal_amount` через `deal_amount_field_mapping`.** Для каждой карточки сумма определяется так:
   - Если задан `deal_amount_field_mapping` — берём `mapping[card.board_id]`, получаем `field_id`, читаем значение из `custom_field_values`.
   - Если маппинг не задан, но задан `deal_amount_field_id` — используем единое поле.
   - Если ничего не задано — `deal_amount = NULL` (режим `count`).

4. **Линеаризация колонок нескольких досок.** Подробности алгоритма линеаризации — в `data-model/multi-board-unification.md`, раздел 1.

Результат — таблица, где каждая строка = одно пребывание одной сделки на одном этапе: когда вошла, когда ушла, сколько часов провела, какой это по счёту визит, куда ушла дальше, `board_id` карточки, `deal_amount` (по маппингу или общему полю).

### Шаг 4. Определить когорту

Для каждой карточки находим дату самого первого входа на любой этап воронки. Если эта дата попадает в выбранный период — сделка входит в когорту. Конечная дата периода включительна (то есть если период до 30 июня, сделки, вошедшие 30 июня, тоже попадают).

Когорта определяется по записям `funnel_deal_history`, которые теперь содержат карточки **со всех досок пространства**. Карточка с доски «Лиды», впервые вошедшая на этап 20 мая, попадает в майскую когорту — независимо от того, что потом она прошла через этапы доски «Сделки».

### Шаг 5. Посчитать метрики по каждому этапу

Для каждого этапа воронки, по сделкам из когорты:

**Количество сделок на этапе.** Считаем уникальные сделки из когорты, которые побывали на этом этапе. Если сделка была на этапе дважды (вернулась) — считается один раз.

**Сумма сделок на этапе (только `metric_mode = 'amount'`).** Суммируем суммы сделок, которые побывали на этапе. Каждая сделка учитывается один раз, берётся сумма из первого визита. Сделки без суммы не включаются в сумму. Сумма берётся с учётом маппинга по `board_id` карточки.

**Средний чек (только `metric_mode = 'amount'`).** Общая сумма делится на количество сделок, у которых сумма заполнена. Если ни у одной сделки нет суммы — метрика не определена.

**Конверсия в следующий этап.** В знаменателе — все сделки из когорты, побывавшие на этом этапе. В числителе — те из них, кто также побывал на любом более позднем этапе (по глобальному `sort_order`). Сделки, которые всё ещё сидят на этом этапе и не сдвинулись — попадают в знаменатель, но не в числитель (консервативный подход). Если на этап не вошла ни одна сделка — конверсия не определена (не ноль процентов, а именно «нет данных»).

> **Пример 1 — обычная ситуация.** За июнь на «Квалификацию» вошли десять сделок. Из них шесть перешли на «Встречу», одна перескочила сразу на «Предложение» (минуя «Встречу»), а три так и остались на «Квалификации». Знаменатель = десять (все, кто побывал). Числитель = семь (шесть ушли на «Встречу» + одна ушла сразу на «Предложение» — обе считаются как «прошла дальше»). Три оставшиеся сидят на этапе — они в знаменателе, но не в числителе. Конверсия = семь из десяти = семьдесят процентов.
>
> **Пример 2 — возврат и повторный проход.** Сделка «Сбербанк» прошла «Встречу», ушла на «Предложение», затем вернулась на «Встречу», затем снова перешла на «Предложение». Для конверсии «Встречи» эта сделка считается **один раз** в знаменателе (уникальная карточка) и **один раз** в числителе (хотя бы раз прошла дальше). Не важно, сколько раз она возвращалась — факт прохождения дальше зафиксирован.

**Конверсия в победу (win rate от этапа).** Аналогично предыдущей, но в числителе — сделки, которые в итоге перешли в колонку победы. Победа учитывается независимо от даты.

**Среднее время на этапе.** Среднее арифметическое времени пребывания на этапе, в днях. Считается только по завершённым пребываниям — сделка должна была покинуть этап. Сделки, ещё находящиеся на этапе, не включаются. Если сделка была на этапе дважды — оба пребывания считаются как отдельные наблюдения. Время календарное (включая выходные).

**Медианное время.** Аналогично среднему, но берётся медиана. Менее чувствительна к выбросам.

**Зависшие сделки (snapshot).** Это snapshot-метрика. Ищем сделки, которые на конец периода всё ещё сидят на этом этапе и находятся там дольше установленного порога. Порог задаётся в настройках воронки для каждого этапа. Если порог не задан — берётся удвоенное медианное время на этапе за последние три месяца. Если медианы нет — порог четырнадцать дней.

**Коэффициент потерь (drop-off rate).** В знаменателе — все сделки из когорты, побывавшие на этапе. В числителе — те из них, кто ушёл в проигрыш именно с этого этапа.

---

### Шаг 6. Посчитать метрики всей воронки

**Общая конверсия.** Количество выигранных сделок из когорты делим на размер когорты. Не может превысить сто процентов. Для свежих когорт будет ниже реальной — часть сделок ещё в процессе.

**Средний цикл сделки.** Среднее время от первого входа в воронку до победы, в днях. Считается только по выигранным сделкам когорты. Если нет выигранных — метрика не определена.

**Объём воронки (pipeline value, snapshot).** Суммарная стоимость всех сделок, которые на конец периода находятся на каком-либо этапе воронки **любой из досок пространства**. Независимо от когорты — это «фотография» воронки. При `metric_mode = 'count'` — количество сделок на этапах. Сумма берётся с учётом маппинга полей по доскам.

**Взвешенный объём воронки (snapshot, только `metric_mode = 'amount'`).** Каждая активная сделка умножается на вероятность победы, привязанную к её текущему этапу. Вероятности задаются в настройках воронки. Если не заданы — используется линейная интерполяция по позиции этапа (первый этап из пяти — двадцать процентов, второй — сорок, и так далее).

**Скорость воронки (velocity, только `metric_mode = 'amount'`).** Показывает, сколько денег «производит» воронка в день. Считается как: количество сделок в когорте, умноженное на конверсию когорты, умноженное на средний чек выигранных сделок, делённое на средний цикл сделки в днях. Результат — рублей в день. Если цикл равен нулю или не определён — метрика не определена.

**Потерянные сделки.** Количество сделок из когорты, перешедших в проигрыш. Дополнительно — разбивка по последнему этапу, на котором сделка находилась перед проигрышем, с указанием количества и суммы потерь на каждом этапе.

## Обработка особых случаев

### Создание карточки сразу в колонке

Если карточка создана сразу в колонке, которая является этапом воронки (не перемещалась откуда-то, а была создана там) — это считается входом на этап, время входа равно времени создания.

### Перескок этапов

Сделка перешла из «Квалификации» сразу в «Предложение», минуя «Встречу». В этом случае запись создаётся только для «Предложения» — этапа, на который сделка реально попала. Для «Встречи» записи нет — сделка туда не входила. При этом конверсия «Квалификации» учитывает, что сделка прошла дальше (попадает в числитель). Конверсия «Встречи» этой сделкой не затрагивается, потому что сделка на «Встрече» не была.

**Мультидосочный перескок:** Сделка может перескочить этапы не только внутри одной доски, но и между досками. Например, если этапы 1-3 на доске «Лиды», а этапы 4-6 на доске «Сделки», сделка может быть создана на доске «Лиды» на этапе 1, а затем перемещена напрямую на доску «Сделки» на этап 5 (минуя этапы 2-4). Логика аналогична: запись создаётся только для этапа 5.

### Возврат на предыдущий этап

Сделка прошла «Встречу», перешла на «Предложение», затем вернулась на «Встречу», потом снова на «Предложение». В этом случае:
- **Для количества сделок:** сделка считается на «Встрече» один раз (уникальная карточка, не важно сколько визитов).
- **Для суммы:** берётся из первого визита, один раз.
- **Для времени:** каждый визит считается как отдельное наблюдение — два пребывания на «Встрече» дадут два значения в среднее.
- **Для конверсии:** сделка попадает в числитель, если хотя бы один раз прошла дальше.

### Перемещение в колонку, не входящую в воронку

Сделка находилась на «Встрече», менеджер перетащил её в колонку «На паузе», которая не является ни этапом воронки, ни победой, ни проигрышем. Пребывание на «Встрече» закрывается (фиксируется время выхода). Сделка считается «покинувшей воронку», но не потерянной. Она не учитывается в числителе конверсии «Встречи» (не прошла дальше). Она не попадает в pipeline (не на этапе воронки). Если потом менеджер вернёт её на какой-то этап — это будет новый вход.

### Проигранная сделка вернулась в воронку

Сделка ушла в «Проигран», но потом менеджер вернул её на «Встречу» (клиент передумал). Итоговый статус сделки определяется по последнему состоянию. Раз она вернулась в воронку — она больше не считается потерянной.

### Сделка без суммы

Учитывается во всех количественных метриках (количество сделок, конверсии). Не учитывается в денежных метриках (сумма, средний чек, pipeline, взвешенный pipeline, velocity). Если все сделки без суммы — денежные метрики отображаются как «нет данных» (не ноль). В режиме `metric_mode = 'count'` денежные метрики не отображаются вовсе.

### Сделка создана сразу на промежуточном этапе

Карточка создана сразу в «Предложении», минуя «Квалификацию» и «Встречу». Когорта определяется по дате входа на «Предложение» — это её первый вход в воронку. На «Квалификации» и «Встрече» этой сделки нет — она не учитывается там ни в чём. В общую конверсию попадает нормально (плюс один в знаменатель, потому что вошла в воронку). Это бывает при горячих лидах от руководства, при импорте из другой системы, или когда менеджер создаёт карточку сразу на нужном этапе.

### Одновременные перемещения (массовая операция)

Менеджер выделил несколько карточек и перетащил их разом. Все перемещения получают одинаковое время. Порядок обработки определяется по внутреннему идентификатору записи перемещения.

### Пустой период

За выбранный период не было ни одной сделки, впервые вошедшей в воронку. Все когортные метрики равны нулю или не определены. Snapshot-метрики (pipeline, зависшие) показываются как обычно — они не зависят от когорты. Отображается сообщение «Нет данных за выбранный период».

### Воронка с одним этапом

Допустимо. Конверсия «этап — следующий этап» не определена (нет следующего). Конверсия в победу считается нормально. Все остальные метрики работают.

### Архивные карточки

По умолчанию архивные карточки исключаются из расчёта. Пользователь может включить их через фильтр. Удалённые карточки учитываются, если их история перемещений сохранена в базе.

### Сделка на доске, не входящей в конфигурацию

Если карточка находится на доске, которая принадлежит пространству, но не входит в `board_ids` конфигурации — карточка **игнорируется**. Это может произойти, когда пользователь вручную исключил доску из воронки через настройки. В режиме Best Guess все неархивные доски включены автоматически.

## Что отдаёт API

### Основной запрос

Пользователь передаёт: идентификатор воронки (`funnel_config_id`) или пространства (`space_id`), начало и конец периода, опционально — режим метрики (`metric_mode`). Опционально — фильтры по ответственным, источникам, тегам, диапазону суммы, архивности. Опционально — даты периода для сравнения.

**Определение конфигурации:** Если передан `space_id` (без `funnel_config_id`) — бекенд ищет сохранённый конфиг. Если не найден — запускает Best Guess и возвращает данные вместе с алертами. Подробности — в `data-model/api-spec.md`, раздел 2.1.

### Что возвращается

Ответ состоит из четырёх блоков.

**Мета-информация:** идентификатор и название воронки, идентификатор и название пространства (`space_id`, `space_name`), массив досок (`board_ids`, `boards[]`), режим метрики (`metric_mode`), флаг автоконфигурации (`auto_generated`), даты периода, общее количество сделок.

**Данные по каждому этапу:** идентификатор и название колонки, порядковый номер этапа, количество вошедших сделок, общая сумма, количество сделок с суммой и без суммы, средний чек, конверсия в следующий этап, конверсия в победу, коэффициент потерь, среднее и медианное время на этапе в днях, количество сделок на этапе сейчас, количество зависших, порог зависания.

**Сводка по всей воронке:** общая конверсия, количество вошедших, выигранных, проигранных и в процессе, средний и медианный цикл сделки, объём воронки и количество активных сделок (с указанием сколько без суммы), взвешенный объём, скорость воронки, средний чек выигранных, разбивка потерь по этапам (для каждого этапа — количество и сумма потерянных).

**Алерты автоконфигурации:** массив `BestGuessAlert[]`. Пустой, если конфигурация сохранённая. Содержит алерты Best Guess, если конфиг автосгенерирован.

### Drill-down

Отдельный запрос для получения списка сделок на конкретном этапе. Поддерживает пагинацию, сортировку и фильтрацию по статусу сделки (все, в процессе, выиграны, проиграны). Принимает `funnel_config_id` или `space_id` по той же логике, что и основной запрос. Каждая сделка в ответе содержит `board_id` и `board_name` — доску, которой принадлежит карточка.

### Сравнение периодов

Если пользователь указал период сравнения, бекенд считает все метрики для обоих периодов (это две разные когорты) и для каждой метрики возвращает:
- Абсолютную разницу (текущее значение минус предыдущее).
- Относительную разницу в процентах.
- Направление тренда: «растёт», если относительная разница больше или равна пяти процентам; «падает», если меньше или равна минус пяти процентам; «стабильно» в остальных случаях.

Автоматический расчёт предыдущего периода: если текущий период — с первого по тридцатое июня (тридцать дней), то предыдущий — тридцать дней до первого июня (со второго по тридцать первое мая).

---

## Правила отображения: ноль и «нет данных»

Ноль и «нет данных» — это разные вещи.

**Ноль** — метрика определена, её значение равно нулю. Пример: конверсия при десяти сделках, из которых ни одна не прошла дальше — это ноль процентов.

**Нет данных** — метрику невозможно рассчитать. Пример: конверсия при нуле сделок на этапе — это «нет данных», а не ноль. Средний чек при полном отсутствии сделок с суммой — тоже «нет данных». На фронте отображается как прочерк.

**При `metric_mode = 'count'`:** все денежные метрики (сумма, средний чек, pipeline value, weighted pipeline, velocity) не рассчитываются и не возвращаются (или возвращаются как `null`). Количественные метрики (количество сделок, конверсии, время) работают без изменений.

---

## Цвет дельт при сравнении периодов

Для большинства метрик рост — хорошо (зелёный), падение — плохо (красный). Но есть «инвертированные» метрики, где всё наоборот.

**Рост = хорошо (зелёный):** количество сделок, сумма сделок, средний чек, конверсия, win rate, скорость воронки, объём воронки.

**Рост = плохо (красный):** среднее и медианное время на этапе (сделки стали медленнее), зависшие сделки (их стало больше), цикл сделки (стал длиннее), потерянные сделки (их стало больше), коэффициент потерь.

Бекенд возвращает для каждой дельты не только число, но и оценку: «позитивно», «негативно» или «нейтрально» — с учётом этой инверсии.

## Производительность

При малом объёме данных (до пяти тысяч перемещений на пространство) — считать всё на лету из истории перемещений.

При среднем объёме (от пяти до пятидесяти тысяч) — материализовать историю прохождения сделок через воронку, агрегаты считать на лету.

При большом объёме (более пятидесяти тысяч) — полная материализация всех промежуточных данных.

**Увеличение объёма при мультидосочной воронке:** В multi-board модели объём данных в `card_movements` и `funnel_deal_history` пропорционально растёт количеству досок (обычно x2-5 по сравнению с single-board). Пороги выше применяются к суммарному объёму по всему пространству.

При каждом перемещении карточки: пересчитать историю только для этой конкретной карточки. Инвалидировать кеш отчёта.

Кеш основного запроса — пять минут. Инвалидация при любом перемещении карточки на **любой доске пространства**.

Кеш автоконфигурации — пять минут (ключ: `space_id`). Инвалидация при: добавлении/удалении доски, изменении колонок, изменении кастомных полей.

Ожидаемое время ответа: до двухсот миллисекунд для малых объёмов, до пятисот миллисекунд при материализации. Автоконфигурация (Best Guess) — до тридцати миллисекунд (2-3 простых SQL-запроса).

## Валидация запросов

Бекенд должен проверять входные данные и возвращать понятные ошибки:
- Не указан ни `funnel_config_id`, ни `space_id` — ошибка «обязательный параметр не указан».
- Начало периода позже конца — ошибка «некорректный диапазон дат».
- Период длиннее года — ошибка «слишком широкий диапазон».
- Минимальная сумма больше максимальной — ошибка «некорректный диапазон суммы».
- Воронка (или пространство) принадлежит другой компании — ошибка доступа.
- Воронка не найдена (по `funnel_config_id`) — ошибка «не найдено».
- Пространство не найдено (по `space_id`) — ошибка «не найдено».
- В пространстве нет досок — ошибка `NO_BOARDS_IN_SPACE` (422).
- Все колонки имеют тип done — ошибка `ALL_COLUMNS_DONE` (422).
- Указана только одна дата периода сравнения — ошибка «неполный период сравнения».
- Недопустимое значение `metric_mode` — ошибка `INVALID_METRIC_MODE`.

## Чеклист для разработчика

### Автоконфигурация (Best Guess)
- Реализовать алгоритм Best Guess: сбор досок -> линеаризация колонок -> определение won/lost -> определение поля суммы.
- При отсутствии сохранённого конфига — автогенерация (не ошибка).
- Возвращать алерты при неоднозначной автоконфигурации.
- Конфиг автоконфигурации не сохраняется в БД; флаг `auto_generated = true`.
- Кешировать автоконфигурацию по `space_id` с TTL пять минут.

### Мультидосочная логика
- Загрузка досок пространства по `board_ids`, сортировка: `row_sort_order ASC, sort_order ASC`.
- Все запросы к карточкам: `board_id = ANY(:board_ids)` вместо `board_id = :id`.
- Глобальная линеаризация этапов: сквозной `sort_order` через все доски.
- Разрешение одноимённых колонок: суффикс с названием доски (`multi-board-unification.md`, раздел 1.3).
- Маппинг поля суммы: `deal_amount_field_mapping[board_id]` > `deal_amount_field_id` > null.

### Режим метрики
- Поддержать два режима: `amount` (по сумме) и `count` (по количеству).
- При `metric_mode = 'count'` — не рассчитывать денежные метрики (сумма, средний чек, pipeline value, weighted pipeline, velocity).
- При `metric_mode = 'amount'`, но поле суммы не задано — автоматический fallback на `count`.

### Когортный расчёт и метрики
- Реализовать построение `funnel_deal_history` из истории перемещений карточек **всех досок**.
- Добавить `board_id` в каждую запись `funnel_deal_history`.
- Реализовать определение когорты по дате первого входа на любой этап воронки в рамках выбранного периода.
- Для когортных сделок показывать все этапы, включая перемещения, которые произошли за пределами выбранного периода.
- Количество и сумму считать по уникальным сделкам, время — по каждому визиту отдельно.
- Snapshot-метрики (pipeline, зависшие) считать по состоянию на конец периода, независимо от когорты.
- Не определённые метрики возвращать как «нет данных», определённые нулевые — как ноль.
- Учитывать инвертированную логику оценки для метрик времени, зависших, цикла и потерь.
- Обработать все особые случаи: перескоки этапов (включая кросс-board), возвраты, создание на промежуточном этапе, уход в колонку вне воронки, возврат из проигрыша.

### Конкурентность и целостность данных
- POST /config: использовать `INSERT ... ON CONFLICT (space_id, company_id) DO NOTHING` + `SELECT` для upsert-поведения.
- Возвращать `201` при создании нового конфига, `200` с `already_existed = true` при обнаружении существующего.
- Ошибка `409 CONFIG_EXISTS` удалена из API-контракта.
- Реализовать `compute_structure_hash(space_id)`: SHA-256 от отсортированных `board_ids + column_ids + custom_field_ids`.
- `GET /auto-config`: возвращать `structure_hash` в ответе.
- `POST /config`: при наличии `expected_structure_hash` — вычислить текущий хеш и сравнить. При несовпадении — `409 STRUCTURE_CHANGED` с новым auto-config.
- Порядок проверок в POST /config: валидация (400) -> доступ (403) -> structure_hash (409) -> upsert (201/200).

### Инфраструктура
- Кеширование с инвалидацией при перемещении карточки на любой доске пространства.
- Кеширование автоконфигурации с инвалидацией при изменении структуры досок.
- Мультитенантность: все запросы через идентификатор компании.
