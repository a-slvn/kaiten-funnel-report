# Варианты упрощения UX настройки графика «Воронка продаж»

## Контекст задачи

### Текущее состояние

При первом открытии отчёта «Воронка продаж» пользователь видит Empty State с кнопкой «НАСТРОИТЬ ГРАФИК». По нажатию открывается диалог (`FunnelSetupDialog`), где:

1. Показаны все доски пространства с toggle вкл/выкл для каждой
2. Для каждой включённой доски — список столбцов, каждый с toggle и выбором роли из Select:
   - **Этап воронки** (серый) — промежуточный этап
   - **Успешный результат** (зелёный) — сделка выиграна
   - **Проиграно** (красный) — сделка проиграна

### Проблемы текущего флоу

| Проблема | Описание |
|----------|----------|
| Много решений одновременно | Пользователь должен сразу решить: какие доски включить, какие столбцы включить, и какую роль присвоить каждому. Три уровня решений в одном экране |
| Нет подсказок «по умолчанию» | Все столбцы помечены как «Этап воронки». Пользователь должен сам догадаться, что последние столбцы обычно — это Won/Lost |
| Toggle на каждый столбец избыточен | CRM-доска обычно имеет 3-7 столбцов. Для типичной воронки нужны все столбцы. Зачем отключать отдельные? |
| Нет визуальной обратной связи | Пользователь не видит, как будет выглядеть воронка до нажатия «Готово» |
| Терминология «роль столбца» неочевидна | Руководители отделов продаж мыслят этапами воронки, а не «ролями столбцов» |

### Жёсткие ограничения

- Пользователь **обязан** вручную указать, какой столбец означает «Успешный результат» и какой — «Проиграно». Автоматическое определение невозможно
- Доска может определяться автоматически из контекста перехода
- Столбцы доски известны заранее (приходят из API)
- Типичная CRM-доска: 3-7 столбцов

### Целевая аудитория

Руководители отделов продаж. Не технические пользователи. Мыслят категориями бизнес-процессов: «Квалификация -> Встреча -> Предложение -> Оплата».

## Анализ конкурентов: как решают задачу настройки

### Pipedrive

- Won/Lost — это **статус сделки**, а не этап pipeline. При создании pipeline пользователь задаёт только промежуточные этапы (drag-and-drop порядок). Финальные статусы (Won, Lost) — системные и предопределены
- Результат: пользователю **не нужно** вручную указывать Won/Lost — это сделано за него на уровне архитектуры
- Подход: минимум настройки, максимум соглашений по умолчанию

Источник: [Pipedrive Knowledge Base](https://support.pipedrive.com/en/article/how-can-i-customize-my-pipeline-stages)

### HubSpot

- По умолчанию pipeline содержит 7 этапов, последние два — **Closed Won** (100%) и **Closed Lost** (0%) — системные
- При создании нового pipeline — шаблон с предзаполненными этапами, включая Won/Lost
- Пользователь может переименовать, но не может удалить Won/Lost
- Подход: разумные defaults + минимальная кастомизация

Источник: [HubSpot Knowledge Base](https://knowledge.hubspot.com/object-settings/set-up-and-customize-your-deal-pipelines-and-deal-stages)

### amoCRM

- Два **системных этапа** — «Успешно реализовано» и «Закрыто и не реализовано» — являются обязательными. Их нельзя удалить, можно только переименовать
- При создании воронки пользователь добавляет только промежуточные этапы. Won/Lost уже на месте
- Подход: системные этапы «из коробки», zero-configuration для Won/Lost

Источник: [amoCRM настройка воронки](https://nova-amocrm.ru/articles/sales-funnel-setup/)

### Monday CRM

- Pipeline стадии — это значения Status Column. «Won» и «Lost» — предопределённые значения в шаблоне
- Funnel Chart widget автоматически использует Status Column для визуализации
- Подход: шаблон с разумными defaults, визуальная настройка drag-and-drop

Источник: [Monday.com Support](https://support.monday.com/hc/en-us/articles/360013348719-Sales-pipeline-management-with-monday-CRM)

### Bitrix24

- Стадии сделки настраиваются в справочнике. Есть системные финальные стадии
- При создании воронки предлагается набор стадий «из коробки»
- Подход: преднастроенные стадии с возможностью полной кастомизации

Источник: [Bitrix24 Справка](https://helpdesk.bitrix24.ru/open/20732764/)

### Вывод из конкурентного анализа

**Все 5 конкурентов** решают задачу одинаково: Won и Lost — это предопределённые системные состояния, а не пользовательская настройка. Пользователь настраивает только промежуточные этапы.

В Kaiten ситуация иная: столбцы доски не имеют предопределённых ролей, поэтому пользователю нужно явно указать, что есть что. Это архитектурное ограничение, которое мы должны сгладить на уровне UX.

## Вариант 1: Inline-разметка с цветными зонами

### Описание

Заменить Select-выпадашки на визуальное деление столбцов в три зоны с drag-and-drop между ними. Пользователь видит три горизонтальные области: «Этапы воронки» (серая), «Успешный результат» (зелёная), «Проиграно» (красная). Столбцы доски отображаются как перетаскиваемые чипы, которые нужно распределить по зонам.

### Флоу

```
+----------------------------------------------------------+
|  Настройка воронки                                        |
|                                                           |
|  Доска: [Продажи v]        (Автоматически из контекста)   |
|                                                           |
|  Распределите столбцы по зонам:                           |
|                                                           |
|  ┌─ ЭТАПЫ ВОРОНКИ (серый фон) ──────────────────────────┐ |
|  │  [Квалификация] [Встреча] [Предложение] [Переговоры] │ |
|  └──────────────────────────────────────────────────────-┘ |
|                                                           |
|  ┌─ УСПЕШНЫЙ РЕЗУЛЬТАТ (зелёный фон) ──────────────────-┐ |
|  │  [Закрытие]                                           │ |
|  └──────────────────────────────────────────────────────-┘ |
|                                                           |
|  ┌─ ПРОИГРАНО (красный фон) ───────────────────────────-┐ |
|  │  [Потеряно]                                           │ |
|  └──────────────────────────────────────────────────────-┘ |
|                                                           |
|  Подсказка: перетащите столбцы в нужную зону              |
|                                                           |
|                            [Отмена]  [Построить воронку]  |
+----------------------------------------------------------+
```

### Начальное состояние (smart defaults)

- Все столбцы помещаются в зону «Этапы воронки»
- Если название последнего столбца содержит слова-маркеры («закры», «оплат», «won», «выигр», «успеш»), он автоматически помещается в зону «Успешный результат»
- Если название столбца содержит («потер», «отказ», «lost», «проигр», «отклон»), он помещается в «Проиграно»
- Пользователь корректирует при необходимости

### Плюсы

| Преимущество | Пояснение |
|-------------|-----------|
| Визуальная ясность | Три цветные зоны сразу дают понять, что нужно сделать |
| Drag-and-drop интуитивен | Руководители привыкли к канбан-доскам, жест «перетащить» для них естественен |
| Меньше решений | Нет toggle-переключателей на каждый столбец. Если столбец не нужен — просто не перетаскивай |
| Smart defaults | Эвристика по названиям столбцов может в 70-80% случаев угадать Won/Lost |

### Минусы

| Недостаток | Пояснение |
|-----------|-----------|
| Drag-and-drop на мобильных | Перетаскивание плохо работает на тач-устройствах. Нужен fallback |
| Сложность реализации | Drag-and-drop с зонами — нетривиальная фронтенд-задача |
| Не масштабируется для multi-board | Если нужно настроить несколько досок — экран станет длинным |
| Эвристика может ошибаться | Smart defaults по названиям — ненадёжные, пользователь может быть сбит с толку ложным срабатыванием |

### Сложность реализации

**Средняя.** Drag-and-drop на React (dnd-kit или react-beautiful-dnd). Эвристика по названиям — простая строковая логика. Основная сложность — качественный drag-and-drop с анимациями и поддержкой accessibility.

## Вариант 2: Пошаговый визард (3 шага)

### Описание

Заменить единый диалог на пошаговый wizard из 3 шагов. Каждый шаг — одно простое решение. Прогресс-бар сверху показывает, где пользователь находится.

### Флоу

**Шаг 1 из 3: Выбор доски**

```
+----------------------------------------------------------+
|  Настройка воронки                   Шаг 1 из 3  [===  ] |
|                                                           |
|  Выберите доску с воронкой продаж:                        |
|                                                           |
|  ( ) Backlog                                              |
|  (*) Продажи                    <- выбрана автоматически  |
|  ( ) Ready for Delivery                                   |
|  ( ) Архив                                                |
|                                                           |
|  Подсказка: выберите доску, столбцы которой               |
|  соответствуют этапам вашей воронки продаж                |
|                                                           |
|                                          [Далее ->]       |
+----------------------------------------------------------+
```

**Шаг 2 из 3: Укажите финальные столбцы**

```
+----------------------------------------------------------+
|  Настройка воронки                   Шаг 2 из 3  [====  ] |
|                                                           |
|  Столбцы доски «Продажи»:                                |
|                                                           |
|  Квалификация -> Встреча -> Предложение -> Переговоры ->  |
|  -> Закрытие -> Потеряно                                  |
|                                                           |
|  Укажите, какой столбец означает:                         |
|                                                           |
|  Успешный результат:  [ Закрытие       v ]  (зелёный)     |
|  Проиграно:           [ Потеряно       v ]  (красный)     |
|                                                           |
|  Все остальные столбцы автоматически станут                |
|  этапами воронки.                                         |
|                                                           |
|                              [<- Назад]  [Далее ->]       |
+----------------------------------------------------------+
```

**Шаг 3 из 3: Проверка и запуск**

```
+----------------------------------------------------------+
|  Настройка воронки                   Шаг 3 из 3  [======] |
|                                                           |
|  Ваша воронка:                                            |
|                                                           |
|  1. Квалификация        (этап)                            |
|  2. Встреча             (этап)                            |
|  3. Предложение         (этап)                            |
|  4. Переговоры          (этап)                            |
|  --------------------------------                         |
|  Закрытие               (успешный результат)              |
|  Потеряно               (проиграно)                       |
|                                                           |
|  Всё верно?                                               |
|                                                           |
|                   [<- Назад]  [Построить воронку]          |
+----------------------------------------------------------+
```

### Плюсы

| Преимущество | Пояснение |
|-------------|-----------|
| Минимальная когнитивная нагрузка | На каждом шаге — одно решение. Шаг 1: какая доска. Шаг 2: какие столбцы Won/Lost. Шаг 3: проверка |
| Понятная ментальная модель | Progress bar даёт ощущение контроля и конечности процесса |
| Экран подтверждения | Шаг 3 даёт уверенность, что настроено правильно — снижает anxiety |
| Ключевой инсайт: остальное — автоматически | Шаг 2 формулирует задачу как «укажите Won и Lost, а остальное станет этапами само». Это переворачивает логику: вместо разметки каждого столбца, нужно указать только 2 |
| Простая реализация | Три последовательных экрана в Dialog — стандартный паттерн |

### Минусы

| Недостаток | Пояснение |
|-----------|-----------|
| Больше кликов | 3 шага вместо 1 экрана. Для опытных пользователей это может раздражать |
| Шаг 1 может быть лишним | Если доска определяется из контекста, зачем её выбирать? |
| Не подходит для переконфигурации | При редактировании существующей настройки wizard раздражает — пользователь хочет быстро поменять одно поле |

### Сложность реализации

**Низкая.** Три последовательных экрана внутри одного Dialog с состоянием `step: 1 | 2 | 3`. Стандартные MUI-компоненты. Stepper для прогресс-бара. Никакой сложной логики.

## Вариант 3: Авторазметка + подтверждение (zero-config)

### Описание

Применить эвристику для автоматической разметки столбцов, а затем показать пользователю результат с возможностью скорректировать. Фокус на «всё уже настроено, просто проверьте и подтвердите».

### Логика авторазметки

```
Правила эвристики (в порядке приоритета):

1. Столбец с ключевыми словами «оплат», «won», «выигр», «успеш», «закры»
   -> роль: won (успешный результат)

2. Столбец с ключевыми словами «потер», «отказ», «lost», «проигр», «провал»
   -> роль: lost (проиграно)

3. Если Won/Lost не найдены по словам:
   - Последний столбец -> won
   - Предпоследний столбец -> lost
   (обратная логика: часто последняя колонка — «Оплата», предпоследняя — «Отказ»)

4. Все остальные столбцы -> stage (этап воронки)
```

### Флоу

```
+----------------------------------------------------------+
|  Настройка воронки                                        |
|                                                           |
|  Мы определили структуру воронки по доске «Продажи»:     |
|                                                           |
|  ЭТАПЫ ВОРОНКИ                                            |
|  ┌─────────────────────────────────────────────────────┐  |
|  │  1. Квалификация                                    │  |
|  │  2. Встреча                                         │  |
|  │  3. Предложение                                     │  |
|  │  4. Переговоры                                      │  |
|  └─────────────────────────────────────────────────────┘  |
|                                                           |
|  ФИНАЛЬНЫЕ СТАТУСЫ                                        |
|  ┌─────────────────────────────────────────────────────┐  |
|  │  Закрытие       [Успешный результат v]  (зелёный)   │  |
|  │  Потеряно       [Проиграно          v]  (красный)   │  |
|  └─────────────────────────────────────────────────────┘  |
|                                                           |
|  Не совпадает? Измените роли столбцов выше.              |
|                                                           |
|                            [Отмена]  [Подтвердить]        |
+----------------------------------------------------------+
```

### Сценарий, когда эвристика не сработала

Если эвристика не нашла ни одного Won/Lost кандидата (нестандартные названия столбцов), показать предупреждение:

```
+----------------------------------------------------------+
|  (!) Мы не смогли определить финальные столбцы.           |
|                                                           |
|  Укажите вручную:                                         |
|                                                           |
|  Успешный результат:  [ Выберите столбец   v ]            |
|  Проиграно:           [ Выберите столбец   v ]            |
+----------------------------------------------------------+
```

### Плюсы

| Преимущество | Пояснение |
|-------------|-----------|
| Минимум действий в best case | Если эвристика угадала — 1 клик на «Подтвердить». Это fastest path to value |
| Обучающий эффект | Пользователь видит предложенную разметку и понимает логику |
| Хороший для повторной настройки | При переконфигурации — всё на одном экране, без wizard |
| Масштабируется | Работает одинаково для доски с 3 столбцами и для доски с 10 |

### Минусы

| Недостаток | Пояснение |
|-----------|-----------|
| Эвристика — чёрный ящик | Пользователь может не понять, почему столбцы разметились именно так |
| Ложные срабатывания | Если столбец «Закрытие переговоров» определится как Won, а это этап — пользователь будет в замешательстве |
| Зависимость от качества эвристики | Если эвристика ошибается часто, доверие к фиче падает |
| Один экран = всё ещё много информации | Для досок с 7+ столбцами экран перегружен |

### Сложность реализации

**Низкая.** Эвристика — набор regex-проверок по названиям. UI — один экран с двумя Select-полями. Сложность только в тестировании эвристики на реальных данных.

## Вариант 4: Inline-маркировка в цепочке (tag-based)

### Описание

Показать столбцы доски в виде горизонтальной цепочки (как pipeline). Пользователь кликает по столбцу и помечает его тегом: зелёным (Won) или красным (Lost). Непомеченные столбцы автоматически считаются этапами. Это паттерн «отметь исключения, остальное — по умолчанию».

### Флоу

```
+----------------------------------------------------------+
|  Настройка воронки                                        |
|                                                           |
|  Доска: Продажи                                           |
|                                                           |
|  Нажмите на столбец, чтобы задать его роль:               |
|                                                           |
|  ┌──────────┐   ┌──────────┐   ┌──────────┐              |
|  │Квалифи-  │-->│ Встреча  │-->│Предло-   │-->           |
|  │кация     │   │          │   │жение     │              |
|  │          │   │          │   │          │              |
|  │  [этап]  │   │  [этап]  │   │  [этап]  │              |
|  └──────────┘   └──────────┘   └──────────┘              |
|                                                           |
|  ┌──────────┐   ┌──────────┐   ┌──────────┐              |
|  │Перегово- │-->│ Закрытие │-->│Потеряно  │              |
|  │ры        │   │          │   │          │              |
|  │          │   │  [won]   │   │  [lost]  │              |
|  │  [этап]  │   │ зелёный  │   │ красный  │              |
|  └──────────┘   └──────────┘   └──────────┘              |
|                                                           |
|  Легенда:                                                 |
|  [этап] — промежуточный этап воронки                      |
|  [won]  — успешный результат (обязательно указать 1)      |
|  [lost] — проиграно (обязательно указать 1)               |
|                                                           |
|  По клику на столбец:                                     |
|  [ Этап воронки | Успешный результат | Проиграно ]        |
|                                                           |
|                            [Отмена]  [Построить воронку]  |
+----------------------------------------------------------+
```

### Поведение по клику на столбец

При клике на столбец появляется inline-popover или сегментированный переключатель с тремя вариантами. Текущая роль подсвечена. Клик по другой роли меняет пометку.

### Плюсы

| Преимущество | Пояснение |
|-------------|-----------|
| Визуальная метафора pipeline | Цепочка столбцов отражает реальный процесс воронки. Пользователь видит свой бизнес-процесс |
| Direct manipulation | Кликаешь по столбцу — меняешь его роль. Прямое взаимодействие, нет промежуточных шагов |
| «Этап по умолчанию» | Не нужно размечать все столбцы — только Won и Lost. Остальные автоматически этапы |
| Наглядная обратная связь | Цвет столбца сразу меняется при назначении роли. Пользователь видит результат мгновенно |
| Масштабирование | Для 3 столбцов и для 7 столбцов работает одинаково хорошо |

### Минусы

| Недостаток | Пояснение |
|-----------|-----------|
| Нестандартный паттерн | Пользователи не привыкли кликать по карточкам для назначения ролей — нужен onboarding |
| Не очевидно, что нужно кликать | Без чёткого affordance (визуальной подсказки) пользователь может не понять, что столбцы интерактивные |
| Multi-board | Для нескольких досок нужен дополнительный переключатель или accordion |
| Горизонтальный скролл | При 7+ столбцах может потребоваться горизонтальный скролл, что неудобно |

### Сложность реализации

**Средняя.** Горизонтальный layout с карточками, popover при клике, анимация смены цвета. Не требует drag-and-drop, но нужна аккуратная работа с layout при разном количестве столбцов.

## Рекомендация

### Основной вариант: Комбинация Variant 2 + Variant 3

Предлагаю объединить сильные стороны Wizard и Авторазметки:

**Шаг 1** — Автоопределение доски (если доска одна или определена из контекста — пропускается). Если досок несколько — выбор из списка.

**Шаг 2** — Авторазметка с подтверждением. Эвристика предзаполняет Won/Lost, пользователь проверяет и при необходимости корректирует через два Select-поля.

**Шаг 3** — (только при первой настройке) Финальное подтверждение с превью воронки.

При **повторной настройке** (редактировании) — сразу показывать Шаг 2, без wizard.

### Почему именно так

1. **Fastest path to value** — если эвристика угадала и доска одна, пользователь видит предзаполненную настройку и жмёт «Подтвердить» (1-2 клика)
2. **Wizard для первого раза** — снижает anxiety для новых пользователей, обучает логике
3. **Single-screen для повторной настройки** — опытные пользователи не проходят wizard заново
4. **Низкая сложность реализации** — никакого drag-and-drop, стандартные MUI-компоненты
5. **Паттерн amoCRM/HubSpot** — «финальные столбцы предзаполнены, скорректируй если надо» — проверенный подход

### Приоритет реализации

| Фаза | Что делаем | Ценность |
|------|-----------|----------|
| MVP | Вариант 3 (авторазметка + 1 экран подтверждения) | 80% ценности, минимум затрат |
| v1.1 | Добавить Wizard-обёртку для first-time experience | +10% ценности, обучающий эффект |
| v1.2 | Добавить Вариант 4 (tag-based) как альтернативный режим для advanced users | +5% ценности, полный контроль |
| Future | Вариант 1 (drag-зоны) если продукт движется к визуальному конструктору воронок | +5%, но высокий cost |

| Вопрос | Контекст | Кому адресован |
|--------|----------|----------------|
| Может ли быть несколько столбцов с ролью Won? | Например, «Оплачено» и «Закрыто без оплаты (подарок)» | Product Owner |
